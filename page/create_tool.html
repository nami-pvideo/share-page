01 · Overview

这份文档干什么用

「纳米P视频」前端的所有创作工具（服装上身、图片翻译、商品替换……）共用同一套接入框架。新增一个工具时，产品 / 运营要做的事可以分成三件——做完这三件，工具就能上线。

🎯
Task 01

想清楚要做什么

从产品 PRD 拿到工具 key 与展示名、明确生图还是生视频；与设计师确认交互稿（Figma）。

产品主导
📋
Task 02

配置云控（CMS）

CMS key + 完整 value：使用指南文案、空状态素材、模型 / 计费、工具专属字段。

运营主导
🚀
Task 03

入口 + 验证 + 提测

配置透出入口 → 通过 skill 产出代码 → 按 第 6 节 验证功能 → 提测上线。

运营主导
✨
你不需要操心的部分

UI 复用、回填 / 重试链路、积分预估、空态展示、登录会员积分……都由现有框架统一处理。开发只需按 开发指南 把工具挂上注册表即可。

📈
越接入越便宜：这套规范的复利效应

每接入一个新工具，模块级 skill 就会沉淀一次（新坑回写到规范、Review 清单加一条、差异表多一列）——下一个工具的接入成本会更低、AI 更熟、出错更少。所以在 第 6 节 验收完成后，建议把这次踩到的新坑反馈给开发，让规范越用越厚。

02 · How to Use

两种使用方式

本文档既可以作为 AI Skill 唤起规范使用，也可以作为普通文档贴路径给 AI。选哪种取决于你是否经常接入新工具。

① 安装 Skill

我们把整套接入规范沉淀成了 Claude Code skill namivideo-creation-tools。推荐用 sparse-checkout 只拉本 skill：

Terminal · zsh 
# 安装到 ~/.claude/skills/namivideo-creation-tools
cd ~/.claude/skills
git clone --filter=blob:none --no-checkout \
  git@github.com:nami-pvideo/skill-hub.git _skill-hub-tmp
cd _skill-hub-tmp
git sparse-checkout init --cone
git sparse-checkout set dev
git checkout main
mv dev ../namivideo-creation-tools
cd .. && rm -rf _skill-hub-tmp

② 验证安装成功

重启 Claude Code，在对话框输入 /，斜杠菜单中应能看到 namivideo-creation-tools 候选项：

③ 调用 Skill

在对话中直接输入 /namivideo-creation-tools，AI 会按 第 5 节「必交付信息」与你逐项确认，然后驱动开发完成对接。

直接贴文档路径

不想装 skill 的话，可以在与 AI 对话时直接把这份文档（或开发版指南）路径贴给它：

Prompt 
请按这份文档接入一个新工具：
/Users/.../nami-video-web/src/features/creation-tools/TOOL_ONBOARDING_FOR_PM.md

AI 会读完文档后按同样的流程与你对接。

💡
为什么推荐 Skill 模式：装上 skill 后，AI 遇到「我想加个新工具」时会主动唤起规范流程；不用每次手动贴路径，也不会漏掉任何关键确认项。

04 · Entry Setup

配置聚合页入口

工具开发完成后，前端路由 /tools/:tool_type 已经通配——但用户从哪里点进来，由聚合页 CMS 决定。新工具接入时你只需填一个字段：「跳转目标」，其余字段（卡片标题 / 封面图 / 分类等）已经由产品 / 设计在工具卡片配置时统一配好。

i
实际只填一项：跳转目标 = 工具 key

下表中带 ★ 的字段虽然都是表单必填项，但除了「跳转目标」外，其它字段（工具名称、分类、封面图、排序、状态等）都已经在工具上架前由产品 / 设计配好，你只要核对 + 把跳转目标填成新工具的 tool_type 字符串即可。

📝 你只需要填「跳转目标」

进入 CMS 后台 → 「创作工具聚合页」 → 新增 / 编辑工具，在表单里：

  • 「跳转类型」选「站内路由」
  • 「跳转目标」填工具 key（tool_type）字符串，例如 viral_replicate。不带 /tools/ 前缀
  • 其余字段（工具名称 / 分类 / 封面图 / 排序 / 状态等）按产品 / 设计提供的内容核对即可，不需要你逐一配置
!
注意事项
  • 「跳转目标」必须与开发注册的 tool_type 完全一致（区分大小写、下划线），否则点击卡片会进入空白页 / 404。
  • 「跳转类型」务必选「站内路由」；选「外部链接」会跳到外部 URL，不走 /tools/:tool_type 路由。
  • 工具一旦上线，tool_type 不建议修改——它同时是路由段、任务筛选 scene 和用户已生成任务的归属字段。
05 · Deliverables

给 AI / 开发的必交付信息

下面 6 项是产品 / 运营在使用 skill 时要主动提供给 AI 的物料。AI 会逐项确认，缺一项不能开始接入。

📦 必交付物料清单

#信息示例说明
1工具 key（tool_type）image_translate路由 / 任务筛选 / 代码常量；建议 snake_case，上线后不可改
2工具展示名图片翻译聚合页卡片标题、编辑区 Header、任务列表标题
3CMS keynamivideo_image_translate与 tool_type 一一对应，命名 namivideo_{语义}
4CMS value 数据结构JSON 草案（见 5.1）使用指南、空状态、模型 / 计费、工具专属字段
5Figma 设计稿链接figma.com/...?node-id=xxx必须定位到编辑区左侧面板节点；交付后 AI 会比对已有交互组件——命中则直接复用，未命中再与你确认交互细节
6介质类型声明生图 / 生视频一句话即可，决定结果区展示形态、提交 API、计费方式与 ConfigModal 形态
🎨
Figma 交付后 AI 会自动判断是否需要新增交互

交付 Figma 链接后，AI 会比对设计稿与现有创作工具的交互组件库：

  • 命中已有交互组件（如模特选择、模板池、语言网格、查看更多抽屉等）→ 直接复用，无需新增 UI 开发。
  • 未命中现有组件 → AI 会就交互细节（数据结构、筛选维度、选中态、空态文案等）与你逐项确认后再开发。

5.1 云控 value 数据结构

💡
不需要你设计解析逻辑

使用指南、空状态素材、CMS 公共字段全部由现有云控通用组件渲染——你只需要把内容按下面结构填进 CMS 即可。

A. 公共字段（所有工具共用，结构固定）

CMS · value · public-fields.json 
{
  "usage_guide_url": "",            // 「使用指南」弹窗的图文/视频链接
  "usage_guide_title": "",
  "usage_guide_description": "",
  "empty_state_material": {         // 任务列表为空时展示的素材
    "title": "生成你的第一张图片",
    "image_urls": [],               // 轮播图
    "media_type": "video",          // "image" | "video"
    "video_url": ""
  },
  "generation": {
    "model_id": "nami-xxx",
    "prompt_template": "...",
    "cost": { "1k": 1.0, "2k": 1.0, "4k": 2.0 },  // 计费矩阵
    "params": { /* 工具专属配置 */ }
  }
}

B. 工具专属字段（与开发约定的业务字段）

图片翻译 · languages + max_language_count 服装上身 · mannequins + 性别/年龄/体型筛选 商品替换 · templates + template_categories + 时长策略
⊘
无数据不兜底

云控未配置 / 拉取失败 / 字段为空时，前端不会用 fallbacks 硬编码假数据——对应区域会展示空态或禁用生成按钮。CMS 数据务必在上线前配齐。

5.3 怎么参考这个示例写自己的 prompt

对照图三的结构，把你自己的工具按下面顺序在一段话里说清楚即可：

  • 「我要接入一个新工具：{工具展示名}，tool_type = {工具 key}」
  • 「CMS key 是 {namivideo_xxx}，value 数据结构如下……」（贴 JSON 草案 / 截图 / CMS 链接）
  • 「Figma 链接：{figma.com/...?node-id=xxx}」
  • 「这个工具是生图 / 生视频工具」
💡
不用一次写全：哪怕你只贴出工具名 + tool_type，AI 也会按门禁机制反问剩下几项；只是一次性写全，能少几轮来回，更快进入开发。
06 · Verification

上线前验收清单

开发自测通过后，由产品 / 运营逐项验收。点击每一项可勾选，进度自动保存在本地浏览器。

✓验收进度
0 / 21

🚪 6.1 入口与导航

  • 聚合页能看到新工具卡片，标题 / 缩略图 / 排序符合预期
  • 点击卡片进入 /tools/{tool_type} 编辑页（无空白页 / 404）
  • 编辑页 Header 显示的工具名与 CMS 配置一致

✏️ 6.2 编辑区

  • 编辑区布局与 Figma 设计稿一致（间距、文案、控件位置）
  • 「使用指南」入口可点开，文案 / 配图 / 视频与 CMS 一致
  • 工具专属交互（语言网格 / 模特选择 / 模板池 / 抽屉等）符合设计稿
  • 必填项未填 / 上传中时，「立即生成」按钮正确置灰
  • 积分预估数字与云控 cost 矩阵对得上

⚡️ 6.3 生成与结果

  • 点击「立即生成」能正常发起任务，未登录会弹出登录 / 会员收银台
  • 结果区任务列表的介质类型正确（生图展示图片，生视频展示视频）
  • 任务列表为空时，空状态素材与 CMS 配置一致
  • 任务卡参数行展示（比例 / 清晰度 / 张数 / 时长等）符合预期

🔄 6.4 重新编辑 / 再次生成 / 失败重试

  • 「重新编辑」能正确回填之前的素材与选项
  • 「再次生成」能用相同参数发起新任务
  • 失败任务「重试」能带原任务 id 重新提交

🛡 6.5 云控容错

  • 故意把 CMS 必填字段清空，前端展示空态 / 禁用而非假数据
  • test / pre / prod 三环境 CMS key 都已配置

🚀 6.6 多环境与发布

  • prod 环境 CMS 数据已配齐 + 聚合页卡片已在 prod 上架

♻️ 6.7 接入后回流（让规范更厚）

验收完成、提测通过后，问自己三个问题——只要有任何一个回答「是」，就反馈给开发把它沉淀进 skill / 接入文档：

  • 这次有没有踩到现有规范没覆盖的坑？（→ 加进「常见注意点」）
  • 有没有遇到 AI 照规范也写错的地方？（→ 在 Review 清单加一条检查项）
  • 这次的提交参数 / 云控字段，跟现有工具差异表对得上吗？（→ 在差异表加一列）
📈
复利来源：每接入一个新工具，规范就更厚一点；下一个工具的接入成本就更低、AI 更熟、出错更少。这是这套接入框架长期价值的核心。
06.5 · Auto Review

AI 自动 Review 保障

AI 产出代码后，会按 skill 内置的 Review 清单逐项自查，确保新增工具不会改动到其它已上线的功能。这是为什么产品 / 运营可以放心提测——每个对接都有一道架构红线兜底。

🛡 模块边界 Review · 改动只落在新工具自己的目录

新工具的代码必须收敛在 src/features/creation-tools/ 下；模块外只允许极少量"白名单"改动。

检查项红 / 绿说明
新增的 Panel / domain / 组件✓ 允许都在 creation-tools/ 内独立目录
修改 router/index.tsx✗ 禁止路由已通配 /tools/:tool_type，不需要改
修改任意已有工具的 Panel / domain✗ 禁止每个工具完全独立，不许"复用 + if 判断"
在 GenerateResult / home 等模块写工具分支✗ 禁止结果区已表驱动，不允许 if (toolType)
修改共享 Store 类定义 / 默认行为✗ 禁止避免影响所有工具的提交 / 回填链路

🏗 架构约束 Review · 是否符合接入规范

  • tool_type 是唯一标识：路由、CMS key、任务 scene、筛选条件全对得上
  • 已在工具注册表登记完整六件套（CMS key / 介质类型 / 解析 / 回填 / 提交 / 任务卡）
  • 介质类型与提交 API、列表刷新参数三者保持一致（生图 ↔ image / 生视频 ↔ video）
  • 公共字段（使用指南 / 空状态 / 计费）走通用解析，未在 Panel 重复实现
  • Panel 使用统一 Hook + Shell 骨架，未自己重新拉云控

📤 提交参数 / 云控 Review · 数据契约不会写错

  • 提交时 scene 与 tool_type 一致；列表刷新参数与介质类型一致
  • 参考素材经合法性过滤，上传中素材不会被提交
  • Prompt 占位符在 domain 层展开，不在 UI 散落拼接
  • 任务的 kwargs 能从历史任务回填，「重新编辑」可还原编辑态
  • 失败 / 取消任务重试时携带原任务 id

⛑ 云控容错 Review · 无数据不兜底

这条是产品 / 运营最该关心的红线——AI 不会用假数据填空：

  • 未新增 fallbacks 文件，云控解析没有硬编码业务数据
  • 云控列表为空时展示空态 / 不可选，而不是假数据
  • 云控必填项缺失时，「立即生成」按钮禁用或给出明确提示
  • 数量 / 上限取自云控（如 max_language_count），不写魔法数
  • 列表项状态非 active 会被自动过滤

👤 会员 / 登录 / 积分 Review · 与已上线工具完全一致

  • 未登录点击生成 → 弹出会员 / 登录收银台
  • 提交中防重复点击（生图 / 生视频各自独立 submitting 状态）
  • Footer 的积分预估使用统一估价函数
  • 未自造一套会员 / 积分 / 登录流程
✓
所有 Review 项都通过后，AI 才会输出 PR 描述模板

PR 中会附带：模块边界结论、架构约束勾叠、提交契约勾叠、是否触及已有工具（若触及附用户确认记录）、tsc --noEmit 类型检查结果、手工链路回归记录。Reviewer 与产品 / 运营都可以基于这份 PR 描述快速判断是否安全合并。

!
这不是替代你的验收

AI Review 只能保障"代码层面没有越界 / 没有破坏架构红线"；用户实际看到的卡片样式、生成效果、文案细节还是要靠 第 6 节验收清单 走一遍。两者配合才完整。

07 · FAQ

常见问答

点击展开 / 收起。

Q1 · tool_type 上线后能改吗？
不建议。它同时是路由段、任务筛选 scene、用户已生成任务的归属。改名会导致用户历史任务在新工具下检索不到。
Q2 · CMS 没配某个字段会怎样？
对应区域展示空态或禁用，不会用假数据兜底。这是为了避免「上线后用户看到假语言 / 假模特」。所以 CMS 数据务必配齐再上线。
Q3 · 我能否复用别的工具的 Panel，加几个 if 判断？
不能。每个 tool_type 都是独立 Panel + 独立 domain，新工具走独立模块。这条是架构硬约束（开发文档 §15.5.0）。
Q4 · 使用指南、空状态、登录 / 会员 / 积分要单独开发吗？
不需要。这些是平台通用能力，已经统一沉淀。运营只需在云控里把内容填好。
Q5 · 生图改成生视频（或反之）能改吗？
理论上可以但代价大——介质类型决定了提交 API、结果区展示、计费方式、ConfigModal 形态。强烈建议在 第 5 节 确认阶段就定下来。
