1. 信息分层模型

AI 在每次会话中拥有的信息，可以清晰地分成三层。理解这三层，是理解 AI 协作行为的基础。

第一层

🌍 世界知识

AI 默认为满

编程通识、语言语法、主流框架、设计模式、网络协议 —— AI 在这里是顶级专家。Go 的 goroutine 调度、React Hooks 规则、RESTful API 设计，随问随答。

第二层

🏢 团队共识

AI 完全空白

项目架构边界、领域术语定义、命名规范、协作约定、历史决策。AI 不知道"订单"和"工单"的区别，不知道为什么 Service 不能调 Handler。

第三层

📋 当前需求

AI 完全空白

本次任务的改动范围、验收标准、上下游影响、限制条件。AI 不知道这次只改 Provider 层，不知道验收标准是跑通 XXX 测试。

🔍 信息传递故障模型

优秀的 AI 协作 = 信息编码优化 + 信道管理——选择性将团队共识和当前需求以 AI 可消费的形式送入上下文窗口。

💡 核心认知：AI 带着第一层满载而来，但第二层和第三层完全是空的。工程师在协作中的核心角色，就是把第二层和第三层补给它。

2. 信息披露的本质

信息从"一个人的脑子里"变成"团队和 AI 都能读取的文档"，经历了一个由模糊到清晰的四层光谱。

🧠 潜意识
连提出者自己都说不清
"这个页面体验还不够好"
→
🤔 个人模糊掌握
单人懂但表达不清
"把那个按钮置灰"
→
💡 个人充分掌握
想清楚了但没落文档
脑子里有完整画面
→
📄 团队共识
写进 PRD / CONTEXT.md
所有人一打开就能读

⬆ 绝大多数 AI 翻车，根因是需求停留在前两层就甩给 AI 了 ⬆

🐕 "狗"与"你家那只金毛"——抽象与具体的距离

抽象级别类比编程中的对应AI 能交付吗？
品种泛指"帮我带只狗回来""改一下这个页面"❌ 会交付一只"随机狗"
品种具体"带一只金毛回来""把支付按钮置灰"⚠️ 方向对了但细节不对
个体具体"带邻居家叫 Lucky 的金毛""当余额不足时，支付按钮 disabled，token=--btn-disabled，hover 提示'余额不足'"✅ 交付你想要的

📐 最小信息披露原则

不是"把所有信息都告诉 AI"，而是"只告诉 AI 它不能自己推断出来的信息"。

✅ AI 自己能推断 → 不需要披露

  • ✓ Go 标准库的用法
  • ✓ 主流框架的常见用法
  • ✓ 通用设计模式
  • ✓ 编程语言语法

❌ AI 推断不出来 → 必须披露

  • ✗ 项目特定的命名约定
  • ✗ 项目特定的架构约束
  • ✗ 项目特有的术语定义
  • ✗ 本次需求的范围和限制
💡 一句话：信息说不清，AI 替你猜；猜对了是你运气好，猜错了是你 review 的工时。下次给 AI 派活前，问自己一句——它在什么地方需要猜？

3. 上下文管理 —— AI 的"工作内存"

上下文窗口不是"能塞多少塞多少"的免费资源，而是要精打细算的稀缺品。

📊 上下文消耗模拟器

拖动滑块，直观感受不同类型内容如何消耗上下文容量（模拟 200K Token 窗口）

使用: 30K / 200K tokens 安全

主动塞（Push）—— 精准控制开局

工程师直接把 PRD 核心段落 + 2-3 个主链路文件 + 本次明确的边界贴进对话。

适用：对话开始时塞入核心上下文，锁定方向，AI 不需要在"我该看什么"上浪费 Token。

注意：只塞 2-3 个核心文件。塞多了反而稀释注意力，AI 在 10 个文件中迷失关键信息。

被动探索（Let Agent Search）—— 可控半径内自主推进

只说目标，让 Agent 自己去 grep、读文件、跑命令，自主搜集它认为需要的信息。

适用：问题定位不明确时的第一轮诊断、跨多个未知文件的开放性问题。

注意：探索本身消耗 Token，可能读到过时文件、把无关模块当主线。

按需取（Pull on Demand）—— MCP 动态拉取

通过 MCP（Model Context Protocol）等接口，让 AI 在需要时自己去拉取信息——API 文档、数据库 Schema、配置中心规则。

适用：可能用到但不一定用的辅助信息，不预占上下文窗口，按需"付费"。

⚡ 混合策略 —— 实战最佳配方

4. Agent vs Workflow —— 两种 AI 模式的信息观

Agent 和 Workflow 最本质的区别，不在于"谁更智能"，而在于对待信息缺失的态度——一个会主动去找，一个必须提前给全。

🔑 核心差异：信息缺失时，谁兜底？

🤖 Agent 模式

"信息不够？我自己去找。"

Agent 的设计哲学是 容忍信息缺口。你只需要给一个方向，缺失的信息由 AI 自主补充——它通过工具调用、文件检索、反问澄清三条路径去拉取未知信息。

✅ 可以缺信息，缺了自己找

✅ 工程师不需要"全知"——想到什么说什么

✅ 适合现实世界：需求从来不是 100% 完整

⚙️ Workflow 模式

"信息不全？直接报错。"

Workflow 的设计哲学是 零容忍信息缺口。每一步的输入必须在编排阶段就定义清楚——缺少任何一个参数，步骤直接中断，没有"自己去补"这回事。

❌ 一步不能缺，缺就报错

❌ 工程师必须预判所有信息需求

❌ 对"意外"零容忍——只能处理预定义分支

🔍 Agent 的三条信息获取通道

当 Agent 发现自己缺少关键信息时，它不会"猜"，而是通过以下三种方式主动拉取——这就是信息披露效率的最大杠杆。

Agent 把"你必须在事前把信息给全"的负担，变成了"你给方向，我负责补齐"的协作模式。

⚡ 信息缺失的两种后果

同样的信息缺口，在 Agent 和 Workflow 中会产生截然不同的结果：

场景Agent 模式的反应Workflow 模式的反应
工程师只说了"改一下订单模块"自动 grep "order"，找到相关文件，读代码理解结构，再定位改动点报错：缺少"要改哪个文件"的输入参数
不知道某个配置项的合法值范围调用 MCP 查询配置中心，或反问工程师"这个值的范围是多少？"报错：配置校验失败，参数超出合法范围
发现"订单"和"工单"两个概念可能混淆读 CONTEXT.md 确认定义，或反问"你指的是用户侧的订单还是系统侧的工单？"不会发现——按预设流程盲走，交付错误结果
需要了解某个历史决策的背景搜索 ADR 存档，或读相关 PRD 文档了解上下文不会尝试——只有预定义的输入，没有"历史决策"这个字段
💡 关键洞察：Agent 的"自主信息拉取"能力，意味着信息披露不需要完美——你不必在事前把一切可能用到的信息都准备好。Agent 会把"信息缺口"从阻塞点变成自动处理的中间步骤。而 Workflow 要求你必须在编排阶段就穷举所有信息——这在现实世界中几乎不可能做到。

🎯 从信息披露角度看选型

核心问题不是"哪个更智能"，而是"你能否在事前穷举所有需要的信息"：

🤖 选 Agent 当...

  • ✓ 你无法在事前穷举所有信息
  • ✓ 信息需要跨文件、跨模块探索
  • ✓ 需求可能在中途出现意外分支
  • ✓ 需要 AI 主动反问、主动验证
  • ✓ 任务是一次性的，不值得花时间编排

⚙️ 选 Workflow 当...

  • ✓ 输入输出在事前完全确定
  • ✓ 不需要任何"探索"或"判断"
  • ✓ 需要定期、无人值守执行
  • ✓ 每一环节需要审计追溯
  • ✓ 对稳定性要求高于灵活性
💡 选型口诀：信息能穷举的用 Workflow，信息有缺口的用 Agent。日常开发 95%+ 的需求天然带有信息缺口——这正是 Agent 的用武之地。

📊 实战对比：同一个任务，两种模式的表现

🏗️ 让 Agent 高效获取信息的四大工程支撑

Agent 的"自主获取信息"不是魔法，它依赖以下工程基础设施：

🛡️ Harness 安全围栏

给 Agent 装"安全围栏"——它能读哪些文件、能跑哪些命令、改代码前是否需要确认。信息获取在可控范围内。

📂 Git Worktree 隔离空间

每个 Agent 独立工作目录，互不干扰。Agent 可以放心探索，不会污染其他人的工作区。

📋 Skill 系统 SOP

brainstorming → plans → execute → verify。告诉 Agent 什么时候该反问、什么时候该拉取、什么时候该验证。

📚 文档中心 长期记忆

CLAUDE.md + CONTEXT.md + ADR + PRD。Agent 的信息获取不是从零开始——持久文档是它的"已知信息基线"。

5. Harness 与 Loop —— 自动化场景下的信息披露

当 AI 不再是你一句我一句的对话伙伴，而是被编排进自动化循环（Loop）中自主运转时，信息披露面临一个全新的挑战：没有人站在旁边随时纠正。Harness 和 Loop 如何协作，让 Agent 在无人值守时依然获得足够且正确的信息？

🔀 从对话到自动化：信息披露的范式跃迁

Prompt 工程、Context 工程和 Harness 工程并不是替代关系，而是层层嵌套、相互托举的共生体。Loop 工程站在四层架构的最顶端——它不关心单次对话的质量，而关心整个系统的自运行能力。信息披露模型也随之从"每次对话手动补"升级为"设计好规则，Loop 自动按规则补"。

从手动变为规则驱动 层层嵌套 相互托举
💡 核心洞察：前三层的职责边界：Prompt 解决"如何对模型说清楚指令"，Context 解决"模型需要知道哪些背景"，Harness 解决"整个 AI 系统如何高可靠、可控地运转"。而 Loop 则把三者的信息流串成一个自驱动闭环——人不再逐条给信息，而是事先定义信息获取规则，Loop 在每一轮自动执行这些规则。

🔄 Loop 五阶段循环：每个阶段的信息披露节点

每一个 AI 编码循环——无论是单 Agent 还是多 Agent——都遵循相同的五阶段循环，直到满足可验证的停止条件。每个阶段都有特定的信息披露关键动作：

💡 Loop 的核心哲学：不要信任模型的上下文窗口作为持久化存储。模型会遗忘、会漂移、压缩信息时会导致约束丢失。每个循环迭代都从一个全新的上下文窗口开始，但需要在实际持久化的内容（STATE.md、Issue 系统、Git 仓库）基础上工作。这就是为什么最原始的 Ralph Loop 只用了一行 while :; do cat PROMPT.md | claude-code; done 就实现了永续自动化——每次循环重新读取 PROMPT.md 和当前代码库状态，完全忽略之前的对话历史，反而解决了所有长对话上下文带来的问题。

🛡️ Harness：自动化场景的信息披露护栏

在 Loop 驱动的自动化场景中，Harness 承担了"确保 Agent 拿到正确信息、不会拿错信息、不乱用信息"的职责。Claude Code、Hermes Agent 和 OpenClaw 三条路线，对"如何在 Loop 中管理信息披露"给出了三种不同的答案。

🔒 Claude Code：信任模型 + 管好边界

信息披露哲学：模型自己能搞定的事就别替它做。Harness 不替模型做决策，只在两个地方卡住：有没有权限做这件事，上下文是不是快溢出了。

信息获取：CLAUDE.md 文件前缀加载 + Auto Memory（模型自己写 Markdown）

权限管控：六种权限模式 · 只读工具自动放行 · 副作用工具每次检查

上下文保护：反应式压缩 · fork 独立实例做摘要 · 3 次失败 circuit breaker

Loop 中的信息流：每轮完整历史交给模型 · 工具执行结果流式返回

📚 Hermes：分层信息积累

信息披露哲学：光给模型看到不够，得帮它在多个时间尺度上积累和召回。

五层记忆：MEMORY.md → SQLite+FTS5 全文搜索 → 8 个外部 memory provider

渐进披露：Skills 分三级加载（名称→内容→附件），模型先扫描再深入

程序性记忆：Agent 完成任务后把方法保存为技能文件，下次复用

Loop 中的信息流：每轮前预取相关记忆 → 每轮后同步对话 → 会话结束抽取长期记忆

🔌 OpenClaw：信息获取作为可插拔服务

信息披露哲学：信息管理的需求会变，与其内置固定方案，不如把接口留好。

Context Engine：可替换运行时插件，四个生命周期点（ingest / assemble / compact / after turn）

会话隔离：DM 共享 session · 群聊按 group 隔离 · cron 每次新 session

多 Agent：每个 Agent 独立 workspace + auth profile + session store

Loop 中的信息流：Gateway 统一协议 → 所有入口收敛到同一套会话管理和工具策略

💡 "Harness 衰变定律"：Anthropic 的核心观察——模型原生能力 ∝ 1 / 所需 Harness 的复杂度。模型越强，需要的 Harness 越薄。这意味着信息披露设计应该为减法而设计：每个 Harness 组件都应该能被安全移除，而不是越堆越厚。模型能自主获取的信息，就不要写死在 Harness 里。把 Harness 收拢在两类长效场景中：① 绝对的合规与业务硬边界（金融风控、权限隔离、Human-in-the-loop）；② 外部环境的互操作系统（API 密钥、数据库连接、状态存储）。

🧩 Loop 六大核心要素的信息披露设计

所有现代 AI 编码工具（Claude Code、Codex 等）都内置了 Loop Engineering 所需的六大核心要素。从信息披露的角度重新审视它们：

要素在信息披露中的角色典型实践
⏰ Automations（自动化） 信息触发源：定义"什么时候该去发现问题"。没有 Automation，信息就躺在那里，Agent 不会主动去看。 定时运行 CI 失败分类、每日 issue 审查、每 2 小时检查性能回归。Claude Code: /loop 定时 + /goal 满足条件才停
🌲 Worktrees（工作树） 信息隔离空间：每个 Agent 看到的是独立副本，互不污染。两个 Agent 并行时不共享文件视角，杜绝"A 改了 B 以为没改"的信息不一致。 每个任务独立 worktree + 独立分支，完成后清理。Codex 和 Claude Code 都原生支持 worktree 级别的隔离
📦 Skills（技能） 信息复用机制：把"每次都要重新解释"的项目知识编码到磁盘上，Agent 按需读取。这是把"你在对话里贴的信息"变成"系统自动注入的信息"。 SKILL.md + 可选脚本 + 参考资料。Agent 发现任务匹配时自动调用对应 Skill。技能是意图的一次性编码，告别"intent debt"
🔗 Connectors（连接器） 信息管道：让 Loop 能触及文件系统之外的信息源——Issue 系统、数据库、CI/CD、Slack。没有 Connectors，Loop 只能看代码，看不到"世界"。 基于 MCP 协议接入真实工具：GitHub Issues、Linear、Slack、PostgreSQL 等
🤖 Sub-agents（子代理） 信息视角分离：让实现者和验证者看到不同的信息子集——实现者只拿到任务所需的文件，验证者拿到完整 spec + 验收标准 + 安全规则。这是对抗"自评自"的信息学策略。 Explore（快速搜索，用 Haiku）→ Implement（全功能）→ Verify（强模型 + 高 reasoning）。验证子代理不共享实现代理的上下文
📝 State（状态） 信息持久化的脊梁：Loop 最核心的设计原则——所有状态存在外部，不在上下文窗口里。模型遗忘一切，但 STATE.md 记住一切。下轮循环从零开始读取。 STATE.md · tasks.json · GitHub Issues · SQLite。记录：已完成 / 进行中 / 尝试过但失败了 / 待处理
💡 Skills 的信息披露经济学：Agent 每轮启动都是"冷启动"——它会用自信的猜测填补任何信息缺口。一个 Skill 就是把项目约定、编码规范、历史教训一次性写在外面，Agent 每次运行都读取，不再重新推导。没有 Skills，Loop 每轮循环都在从零重新推导项目知识；有了 Skills，信息在循环之间复利增长。Hermes 甚至进一步——Agent 完成任务后可以把方法保存为新 Skill，积累 20+ 自创技能后，同类任务速度提升 40%。

🔐 闭环 vs 开环：信息披露的安全边界

Loop 有两种基本类型，信息披露的要求截然不同：

🔒 闭环（Closed Loop）

有明确停止条件，Agent 在有限信息范围内运作。

信息披露要求：

  • 明确的目标（Goal）：精确定义"完成"的样子
  • 充足的上下文：VISION.md + ARCHITECTURE.md + RULES.md
  • 受限的动作：只暴露必要工具，不给多余权限
  • 客观的反馈：测试、lint、类型检查的自动结果
  • 清晰的停止条件：可验证的成功标准

适合：CI 修复、代码审查、测试生成等边界明确的任务

🔓 开环（Open Loop）

持续运行，没有预设终点。需要人类介入来控制信息披露边界。

信息披露要求：

  • 人类审查机制（Human-in-the-loop）：关键节点必须人类确认
  • 自动分类 TRIAGE：完成/需审查/放弃——区分不同信息可靠度
  • 成本上限硬约束：max-iterations + 预算封顶
  • 优先级排序：从问题严重性决定"先看哪个信息"
  • 异常检测：如果输出质量偏离基线，自动降级到闭环

注意：开环的风险极高，中等代码库 50-100 次迭代可烧 ¥500-1000。建议从闭环开始。

🔌 熔断机制（Circuit Breaker）—— 开环的最后一道信息护栏：无人值守的 Loop 必须有硬性停止条件，否则一个陷入"反复尝试同一个错误修复"的 Agent 会悄悄烧光预算。生产级参数参考：最大迭代 15–25 步 · 单轮超时 ~300 秒 · 单次成本预算 ~$2。一个实用技巧——对每轮迭代的动作取哈希指纹，连续 3 次相同立即停止（指纹重复说明 Agent 已陷入死循环，再喂信息也无济于事）。从信息披露视角看，这是一条"负向边界"：当 Agent 不再产生新信息时，及时切断比继续供给信息更重要。Anthropic 的研究员有句话点破本质——"Rubric design is the skill now, the model is the easy part."（设计可验证的成功/停止标准，才是今天真正的工程能力。）
💡 闭环五要素的信息披露视角：闭环之所以"闭"，不是因为 Agent 不获取信息，而是因为信息获取的范围是预先划定好的——它知道自己该看什么文件、不该看什么文件、什么工具可用、什么操作被禁。这种"受限的信息自由"正是 Harness 设计的精髓：不是让 Agent 什么都知道，而是让它在知道得足够的前提下，不乱知道不该知道的。

🎯 Harness + Loop 信息披露设计原则总结

📌 原则 1：状态必须外部化

模型会遗忘，上下文会压缩。所有 Loop 状态（完成/进行中/失败）写入 STATE.md 或 Issue 系统，不在上下文窗口中靠记忆。

📌 原则 2：信息视角必须分离

实现者看到"怎么做"的信息，验证者看到"对不对"的信息。用 Sub-agent + 不同模型 + 独立上下文来强制执行这种分离。

📌 原则 3：Skills 是信息的一次性编码

不要在每个 Loop 迭代中重新解释项目约定。把约定写成 Skill，让 Agent 按需加载——信息在迭代间复利增长。

📌 原则 4：Harness 为减法而设计

模型在进步，今天需要的硬编码约束明天可能是死代码。只把"绝对的合规硬边界"和"外部环境连接器"写死在 Harness 里。

📌 原则 5：从闭环开始，谨慎开环

闭环有明确信息边界——Agent 知道自己该看什么。开环的信息获取是无限度的，必须有成本上限、迭代限制和人工审查节点。

📌 原则 6：Connectors 让 Loop 看到"世界"

没有 MCP Connectors，Loop 只能看到代码文件系统。接入 Issue 系统、CI/CD、Slack，Loop 才能实现"发现问题→修复→PR→通知"全闭环。

💡 终局洞察：人类掌舵，Agent 执行。Loop + Harness 的信息披露设计，本质上就是把"人类每次给 AI 补信息"这件事，从手动操作变成系统规则。你的角色从"每次对话的信息搬运工"升级为"信息获取规则的设计者"——这是一次根本性的工作方式升维。正如 Claude Code 负责人所说："I don't prompt Claude anymore. I have loops running that prompt Claude and figuring out what to do. My job is to write loops."

6. Skill 与信息管理 —— 将 AI 无法感知的信息注入上下文

如果说 Prompt 是"每次对话时告诉 AI 的话"，Context 是"让 AI 在工作过程中看到的东西"，那么 Skill 就是"AI 主动加载到上下文中的、写死在磁盘上的项目知识"。它是信息披露从"手动逐条给"迈向"系统自动注入"的关键一步。

🧠 AI 的信息感知边界

AI 模型有一个根本性的约束：它只能"看到"你放进上下文窗口的东西。模型不会主动知道你的项目用了什么框架、你们的 Git 分支命名规范是什么、团队最近做了什么架构决策——除非有人（或某个机制）把这些信息编码成文本，塞进它的上下文中。

❌ 没有 Skill 的 AI 协作

  • 每次对话都要重新解释项目背景
  • 规范依赖人的记忆——"我刚才说了用 Vitest 吗？"
  • Agent 拿"通用最佳实践"代替"团队约定"
  • 换一个人、换一个会话，知识归零
  • Loop 每轮迭代从零推导项目规则

✅ 有 Skill 的 AI 协作

  • 项目知识一次编写，所有会话自动加载
  • Skill 文件即"团队意志的持久化文本"
  • Agent 按任务意图自动匹配对应 Skill
  • 新人入职 = 一套 Skill 集合 = 团队最佳实践
  • Loop 每轮按需注入 Skill，信息在迭代间复利增长
💡 核心洞察：Skill 的本质是信息披露的自动化编码。它把"每次对话时你要手动贴给 AI 的项目知识"变成了"AI 在看到任务时自己去磁盘上找出来读的东西"。在这个意义上，Skill 不是"让 AI 变聪明"，而是让 AI 能"看到"它本来不可能知道的东西。

📥 Skill 的三级信息加载机制

高效的信息披露不是"把一切塞进去"，而是按需逐步展开。现代 Skill 系统采用三级渐进式加载，在"告诉 AI 足够多"和"不浪费 Token"之间取得平衡：

📛
第一级：名称 + 描述

Agent 启动时加载所有 Skill 的名称和一句话描述。Token 成本极低，但让 Agent 知道"有哪些知识可用"。

📄
第二级：SKILL.md 正文

Agent 判断任务匹配某 Skill 后，按需读取完整指令。包含流程、约束、示例——仅在"匹配时"消耗 Token。

📦
第三级：附带资源

脚本、模板、参考文档。仅在 Skill 被激活后，由 Agent 按需读取——不做预加载。

💡 信息经济学视角：这三级加载实现了 "按需付费"的信息消费模型。第一级全量索引（~50 tokens/Skill），第二级匹配加载（~500-2000 tokens），第三级极少数场景才触发。对比"把项目文档全塞进上下文"加 20000+ tokens，渐进加载在信息覆盖和信息成本之间找到了最优解。

📈 Skill 的信息复利效应

Skill 的价值不是线性的——它在时间维度上产生信息复利。每一次使用 Skill，都是对团队知识的强化而非重新推导：

维度无 Skill有 Skill
会话启动成本 每次对话都要解释项目背景、规范、工具链 Skill 名称索引自动加载，匹配后按需展开完整指令
Loop 迭代效率 每轮从零推导项目规则，消耗 Token 做重复推理 每轮注入 Skill，规则变成"已知"而非"待推理"
知识沉淀 依赖人的记忆和口头传授，离职即流失 Skill 文件即知识库，Git 版本控制，团队共享
Agent 行为一致性 不同会话、不同人的 Agent 行为差异大 同一 Skill 约束下，Agent 行为模式高度一致
自我进化能力 每次踩坑后靠人更新文档（大概率忘记） Agent 完成任务后可把方法保存为新 Skill（如 Hermes 的程序性记忆）
💡 信息复利的工程隐喻：无 Skill 场景下，Agent 每轮 Loop 迭代都在重新交学费——它不知道上次怎么失败的、不知道团队用什么工具、不知道哪些文件不能改。Skill 把"学费"交一次，写成文本，然后所有未来会话和 Loop 迭代共享这份知识。Hermes 实测数据：Agent 积累 20+ 自创 Skill 后，同类任务速度提升 40%。

🔮 展望：模型进化后，流程 Skill 的角色会如何变化？

一个值得正视的判断：随着模型原生能力的提升，今天很多"流程型 Skill"的边际价值必然会下降。这不是 Skill 体系的缺陷——恰恰说明 Skill 是当下模型能力边界的"补丁"，而补丁终究会随着模型进化而被吸收。

📉 会被模型能力吸收的 Skill 类型

  • 流程步骤型：TDD 循环、Code Review 检查清单——模型推理能力增强后，它会"自然"遵循这些最佳实践，不需要显式的 Skill 步骤
  • 格式约束型：输出 JSON schema、Commit message 格式——结构化输出能力内化后，格式 Skill 降级为偏好声明
  • 基础安全规则：SQL 注入检查、XSS 防护——安全推理能力提升后，变为模型内置行为
  • 通用错误处理模式：try-catch 模板、重试逻辑——模型写这些就像今天写 for 循环一样自然

📈 不会消失的 Skill 类型

  • 团队特定约定：分支命名、代码组织、部署流程——模型再强也不可能猜到"你们团队用 __tests__ 还是 spec/ 目录"
  • 外部工具集成：公司内部 CLI、自建 API、定制 CI 脚本——这些都是模型训练数据之外的信息
  • 业务领域知识：风控规则、定价逻辑、特定行业术语——来自业务而非技术，模型无法"推理"出来
  • 组织流程编排：多 Agent 协作模式、审批流程、发布节奏——组织特有的流程设计
💡 判断一个 Skill 是否会被模型吸收的简单法则：问自己——"这个信息，一个足够聪明的模型能自己推理出来吗？"如果能（如 TDD 步骤、错误处理模式），未来可能不需要 Skill。如果不能（如公司 CI 配置路径、团队 ESLint 规则、业务领域模型），Skill 就是永久的信息载体。
🎯 对当下的指导意义：正如 Section 5 中的 Harness 衰变定律 所揭示的——模型原生能力 ∝ 1 / 所需外部规则的复杂度。Skill 设计应当为减法而设计：把"模型学不会的外部信息"写进 Skill，把"模型迟早能自己掌握的流程"留给模型推理。好的 Skill 集合会随着模型进化而自然瘦身，而不是越堆越厚。

🎯 Skill 与信息管理：关键结论

📌 Skill = 信息披露的自动化

打破"每次对话都要手动贴信息"的循环。项目知识从人的短期记忆变成系统的持久文本。

📌 渐进加载 = 信息经济学最优解

名称索引（~50 tokens/Skill）→ 按需加载正文 → 极少数触发附带资源。信息覆盖最大化的同时 Token 成本最小化。

📌 Skill 产生信息复利

知识编码一次，所有未来会话和 Loop 迭代共享。Agent 不再从零推导项目规则。

📌 为减法而设计

把"模型学不会的外部信息"写进 Skill，把"模型迟早能自己掌握的流程"留给推理。Skill 集合应随模型进化而自然瘦身。

7. 新员工心智模型 —— 像带新人一样带 AI

差别往往不在技术层面，而在一个基础认知：你到底是把 AI 当"全知全能的代码生成器"，还是当"能力很强但什么都不懂的第一天入职员工"？

🧑‍💼 类比映射表：新员工的日常 → AI 协作的工程实践

新员工入职需要什么映射到 AI 协作工程实现
入职资料包（项目简介、术语、规范）补齐团队共识层CLAUDE.md + CONTEXT.md + ADR
一张干净的工位和电脑独立的执行环境Git Worktree + 独立会话

师傅的工位旁边可以随时问对话式交互Agent 模式的思考-行动循环

📝 三段式指令模板

好的 AI 任务指令，跟带新员工时的交代几乎一样：

【目标】
给订单模块的 CreateOrder 函数加一个并发锁，防止超卖。

【起点】
- 核心参考文件：orders/service.go 的 CreateOrder 函数（第 45 行）
- 参考实现：inventory/service.go 里的锁实现（完整、线上验证过的写法）
- 领域术语：见 CONTEXT.md 的"订单 → 库存扣减"段

【边界】
- 要改的文件：orders/service.go（只改这一个）
- 不要动的层：Gateway 层、Handler 层、Hook 层
- 不要改 import 部分以外的任何代码
- 超出范围的问题：只提出来，不要直接改

【验收】
跑 go test ./orders/... -run TestCreateOrder -count=10，全部通过即为完成。

🔄 Skill 系统：不让 AI 跳步骤

💡 心智转变的三个触发动作：
① 每次派活前花 30 秒补信息——术语 AI 能不能自己搞懂？文件范围能看出来吗？哪些不能动？
② 接受"AI 会犯错"并建立纠偏机制——犯错不可怕，纠偏成本极低（回滚即修复）
③ 停止期待 AI"自己悟"——假设 AI 什么都不懂，每次需要它懂什么，就说清楚

8. 实战 Checklist —— 上手就能用

不需要一次全部做到。从最下面的一条开始，每一周加一条，一个月后整个协作模式会发生质变。

  • 在项目根目录创建 AGENTS.md，写清：架构分层（哪层能调哪层）、目录约定、禁止事项
  • 创建 CONTEXT.md，定义 3-5 个最容易被 AI 搞混的核心术语
  • 确保 AI 工具配置了自动读取 AGENTS.md
  • 每次派活用三段式指令：目标 + 起点 + 边界
  • 明确告诉 AI 哪些文件/模块不能动
  • 给出验收标准——跑什么命令、哪个测试通过就算完成
  • 安装 git worktree 工作流（git worktree add ../proj-feat feat/xxx）
  • 每个 AI 任务开独立 worktree + 独立会话
  • 不同任务不混在同一个会话里
  • Review AI 代码时，发现系统性问题马上写进 AGENTS.md（如"Provider 改动不触及 Gateway"）
  • 每次 AI 犯错 = 一次文档升级的机会
  • 把常用模式沉淀为 Skill（SOP），让 AI 不跳关键步骤

9. 反模式速查 —— 常见的坑与修正方案

⚠️ 上下文污染八大高发场景

❌ 一次性塞入大量日志

几千行日志只有 2-3 行关键错误，被淹没。AI 逐行分析日志耗尽上下文。

✅ 修正

先用 shell 过滤 ERROR/FATAL，只送有价值上下文

❌ 把整个 README 贴进对话

AI 完全可以自己 read。贴进去分散它对关键代码的注意力。

✅ 修正

让 AI 自己读文件，不要主动贴

❌ 一个会话混做多个任务

旧任务讨论占着上下文，新任务关键信息被稀释。

✅ 修正

不同任务开不同会话，一条会话一条事

❌ 主动塞入"可能用到"的所有文件

10 个文件同时看，AI 在关键信息中迷失。

✅ 修正

只贴 2-3 个核心文件，AI 需要更多自己读

❌ 上下文快满时追加关键信息

工具可能自动压缩历史，追加内容被"挤掉"关键细节。

✅ 修正

感觉上下文快满时，开新会话，总结当前结论作为起点

❌ 超长 Prompt 模板

几百行模板 + 实际任务，上下文窗口已去一小半。

✅ 修正

长期规则放 CLAUDE.md，对话中只写本次直接相关的
