一、权力让渡简史

2023—2026，四次跃迁，三次权力让渡——Prompt → Context → Harness → Loop。

💡 开场钩子：2026 年 6 月，AI 工程圈被两句话同时点燃。Anthropic Claude Code 负责人 Boris Cherny 说：「你不应该再提示编码 Agent 了。你应该设计循环来提示你的 Agent。」几乎同时，Harness Engineering 的概念刷屏技术社区。这两股声浪指向同一个方向——AI 工程范式在一年半内完成了三次跃迁。

这四次跃迁不是在讲技术名词的更替，而是在讲同一件事——每一次「进化」，本质上都是一次权力让渡：人类把越来越多的决策权，从自己手里交出去，交给系统。

年份范式让渡了什么人还保留什么
2023-2024Prompt Engineering「怎么写」人设计提示词措辞和模板——AI 是执行工具，权力主体仍在人手里
2025Context Engineering「信息获取方式」人不再每次手动喂信息，而是构建文档、知识库、检索系统——AI 自主探索和理解上下文
2026 初Harness Engineering「执行环境的控制权」人不再每轮手动控制 Agent，而是设计护栏、约束、反馈循环——Agent 在规则内自主运行
2026 中Loop Engineering「任务流转的决策权」人只定义目标和停止条件，Loop 自动发现→计划→执行→验证→迭代——人在循环之外掌舵

这条弧线在编程领域跑得飞快——2026 年 Claude Code、Codex 已在工程团队里大规模落地，OpenAI Codex 靠 3 人团队 5 个月产出 100 万行代码、合并 1500 个 PR，LangChain 仅优化 Harness 层就让 Terminal Bench 得分从 52.8% 跃升至 66.5%。但这里有一个令人不安的问题：

二、编程为什么赢了

代码库天生就是一个包含解决问题所需大部分信息的环境——这是 AI 在编程领域势如破竹的根本原因。

🏗️ 代码库 = 天然信息富矿

一个 Agent 要改订单模块的防超卖逻辑——它 grep "order"，读 4-5 个文件，跑一遍测试，大概率就搞清楚上下文了。编程领域的信息环境有几个独特属性：

  • 语法和语义确定：编译器不接受歧义。每一行代码的含义精确、可验证、可追溯。
  • 边界清晰：文件、模块、包、依赖关系——整个项目的骨架写在磁盘上，Agent 可以自主探索。
  • 依赖可追溯：import / require / go.mod——外部依赖显式声明，链路完整。
  • 决策文档可部分推导：代码本身记录了"怎么做"，Commit message 记录了"为什么改"，ADR 记录了"为什么选这个方案"。
  • 反馈循环秒级：Lint → 编译 → 测试，一个循环三十秒。AI 改错了，立刻知道。

🏜️ 商业决策的信息环境 —— 荒漠

维度编程领域商业决策 / 广告策划
核心信息在哪代码仓库（结构化文本）人的经验、口头传达、散落文档
信息密度极高——每个文件都是精确的机器指令极低——关键信息在"老张知道"里
可验证性编译通过 / 测试通过 = 机器可判决策好坏可能要数月后才见分晓
反馈速度秒级周或月级别
信息编码形式统一语法、统一仓库Word + Excel + 微信 + 口述 + 会议纪要

编程领域的信息环境：一个模型能靠工具自主探索、逐步逼近真相的封闭宇宙。 商业领域的信息环境：一个模型连"真相在哪"都不知道的开放荒漠。

📊 三层信息模型 —— 编程为什么先天占优

第一层

🌍 世界知识

AI 满载

编程通识、语言语法、主流框架、设计模式、网络协议。AI 在这里是顶级专家。

第二层

🏢 领域共识

编程领域：60-80% 已覆盖

代码本身就是结构化的领域共识，Git 历史是决策记录，架构分层是项目契约。AI 进入一个代码库，不是从零开始——它有一座图书馆。

第三层

📋 当前需求

AI 完全空白

本次任务的改动范围、验收标准、上下游影响、限制条件。需要工程师每次补给它。

🔑 核心洞察：同一个 AI 模型，同样的推理能力，为什么在编程领域一气呵成、在商业领域举步维艰？核心差别不在于 AI 的智力，而在于它"看到"了什么样的信息环境。编程给了 AI 一座图书馆，商业只给了 AI 一堆空白格子。 商业领域的第二层信息几乎空白——领域共识在"老张的经验"里、"上次开会说的"里、"那个 Excel 表你找小王要"里。老张脑子里有 10 年的信息环境，但 AI 一字节也读不到。

三、信息环境 —— AI 能力的核心杠杆

同一个模型，不同的信息环境 = 天壤之别的表现。给 AI 一个好环境，它就能交付好结果；给 AI 一个烂环境，再强的模型也救不回来。

🔍 信息传递故障模型

串联这条链路有两大故障点：编码不足（信息没说清）和 信道过载（有效信号被淹没）。优秀的 AI 协作 = 信息编码优化 + 信道管理。

🧬 模型能力提升的隐藏公式

后训练的本质很大程度上是在让模型熟悉环境。类比：老员工的价值来自口口相传的环境信息——哪个系统容易在月底崩溃、哪个客户有特殊偏好、上次事故的真实原因。这些信息不在任何文档里，在人的脑子里。

模型也一样。一个经过领域后训练的模型，不是在"变聪明"，而是在"变熟悉"——它对这个环境的信息结构越来越了解，不需要每次都从零推导。同样是 Claude，经过编程领域后训练的版本在代码任务上的表现远好于通用版本——不是模型架构变了，是环境信息补上了。

⚡ 环境补全指令 —— 为什么抽象指令也能精准执行

工程师说："给订单模块加防超卖的锁，参考库存那边的写法。"——15 个字，极度抽象。Agent 做了四步：

  1. grep "inventory" → 找到 inventory/service.go 的 MutexLock 实现
  2. grep "CreateOrder" → 找到 orders/service.go 的目标函数
  3. 读 CONTEXT.md → 确认领域术语和链路关系
  4. 读 AGENTS.md → 确认哪些层能改、哪些不能动

最终落地到 LLM 的指令变成了 100+ 个字——信息完备、边界清晰、可验证。

💡 核心价值：信息环境让 Agent 成为人类抽象意图与 LLM 精准执行之间的"翻译层"。信息环境越丰富，人需要的指令就可以越抽象；人越省力，AI 产出的质量反而越高。这是一个反直觉的正循环。

🔄 数据飞轮 —— 谁先抢占用户真实环境，谁就赢

// 飞轮公式 真实使用 → 真实环境交互 → 真实信息积累 → 更好的后训练数据 → 更强的模型表现 → 更多真实使用

编程领域的飞轮已经在转了：GitHub Copilot 有全球最大的代码库交互数据，Claude Code 有 Auto Memory，Cursor 有隐式反馈。但商业领域呢？飞轮还没真正启动。谁先抢占用户在实际工作环境中的使用，谁就更有机会成为商业 AI 领域的赢家。这不是模型能力的竞争，是谁先建起信息环境的竞争。

🎯 WorkBuddy 的独特卡位

腾讯 WorkBuddy 有机会成为在用户的真实系统中构建信息环境的平台：

  • 不是让用户来适应 AI，而是 AI 融入用户已有的工作流——WorkBuddy 嵌在操作系统层面，不是又一个需要用户迁移数据的独立应用
  • AI 看到的不只是代码——还有文档、会议记录、审批流、业务数据、通讯记录。这是编程 Agent 永远接触不到的信息维度
  • 每一次交互，不只是完成一个任务，也是在积累这个组织的信息环境——用户偏好、团队约定、业务规则，在使用中自然沉淀

🔑 关键判断：这轮 AI 竞争的下半场，不是比谁的模型更强——模型能力正在趋同。比的是谁能在用户的实际工作环境中，先建起一套可自我强化的信息环境。 WorkBuddy 在操作系统层面的嵌入优势，是纯 SaaS 工具不具备的护城河。

五、怎么构建

从编程验证过的工程实践出发——这些不是"编程专属"的技巧，而是任何需要为 AI 构建信息环境的领域都能复用的通用范式。

🤖 Agent vs Workflow —— 信息缺口大小决定模式选择

🔄 Agent 模式

"信息不够？我自己去找。"

对信息缺口 → 容忍，自主补全
工具调用 + 反问 + 外部拉取 + 文档注入
使用者 → 想到什么说什么
适合 → 问题定位不明确

⚙️ Workflow 模式

"信息不全？直接报错。"

对信息缺口 → 零容忍，缺就中断
使用者 → 必须事前穷举所有信息需求
适合 → 流程确定、信息可穷举
💡 选择口诀：信息能穷举的用 Workflow，信息有缺口的用 Agent。商业场景 95%+ 的需求天然带有信息缺口。

📡 上下文管理：Push / Explore / Pull 混合策略

信息环境的构建不是"把所有信息都塞给 AI"，上下文窗口是精打细算的稀缺品：

📥 Push（主动塞）

对话开始塞入核心上下文——任务 Brief + 2-3 个关键数据源 + 明确边界。锁定方向。

🔍 Explore（自主探索）

只说目标，让 Agent 自己搜索文件、查询数据库、读取文档。适合问题定位不明确。

📤 Pull（按需拉取）

通过 MCP 等接口，AI 需要时自己拉取——API 文档、数据库 Schema、配置中心规则。不预占上下文窗口。

混合策略实战：前期 Push 定方向 → 中期 Agent 在安全半径内自主探索 → 后期按需 Pull 补细节。

🎯 信息披露的本质：从抽象到具体的距离

"帮我做一下这个季度的广告投放分析" vs "基于 Q2 的投放数据（见 data/Q2-ads.csv），按 ROAS > 2.0 筛选渠道，对比去年同期，输出一页 PPT 摘要"——天壤之别。

抽象级别类比商业场景对应AI 能交付吗？
泛指"帮我带只狗回来""做一下投放分析"❌ 交付随机结果
品种具体"带一只金毛回来""分析 Q2 各渠道 ROAS"⚠️ 方向对了细节不对
个体具体"带邻居家叫 Lucky 的金毛""基于 data/Q2-ads.csv，ROAS > 2.0 的渠道，对比去年同期，输出一页 PPT"✅ 交付你想要的
💡 最小信息披露原则：不是"把所有信息都告诉 AI"，而是"只告诉 AI 它不能自己推断出来的信息"。下次给 AI 派活前，问自己一句——它在什么地方需要猜？

🧩 Skill：让信息环境可编程

前面的章节讲清楚了：信息环境是 AI 能力的核心杠杆。但怎么把信息环境从「人脑子里的事」变成「AI 能稳定读到的事」？Skill 给出了答案。Superpowers 的 14 个 Skill 本质上在做同一件事：把信息注入 AI 的过程工程化——在正确的时间、以正确的粒度、用正确的强制力，把正确的信息塞进 AI 的上下文。

三种工程化手段

🔒 行为约束

把隐性纪律变成显性信息。人知道「先查根因再修 bug」，AI 不知道——Skill 编码了这些工程纪律，在 AI 产生「改一下试试」的念头之前就切断了退路。

🛡️ 合理化拦截

预判 AI 会用什么借口绕过规则（「太简单不需要测试」），提前写好反驳。每次 AI 出现新借口，就补一条反驳——TDD 驱动迭代。

📡 渐进式披露

信息不是一次性全塞进去。Skill 按阶段触发——遇 Bug 加载 debugging，写代码加载 TDD。Frontmatter（触发条件）→ 正文（核心指令）→ 附属文件（按需引用）。

模型进化下，什么 Skill 会消失，什么会留存

Skill 类型模型能学会吗？命运
通用工程纪律（TDD、先查根因）✅ 能被吸收逐渐消失——更强的模型不再需要外部提醒
通用流程模板（brainstorming 九步）✅ 能被吸收逐渐消失——模型原生就知道怎么拆解任务
合理化拦截（「太简单不需要测试」）⚠️ 部分吸收部分留存——模型越强越少狡辩，但总有新借口
领域知识注入（五级回溯法）❌ 学不会永久留存——不在任何公开训练数据里
项目/团队约定（命名规范等）❌ 学不会永久留存——组织私有信息
💡 核心洞察：Skill 的长期价值不在于「教 AI 怎么做」，而在于「告诉 AI 关于这个环境它不可能从训练数据中知道的事」。模型进化让第一层（世界知识）越来越强，但第二层（领域共识）和第三层（当前需求）的信息永远需要人来构建。模型越通用，环境越重要。 Skill 不是模型的补丁，是信息环境的可编程接口。

AI 不是全知全能的代码生成器，而是"能力很强但第一天入职的新员工"。老员工的价值来自口口相传的环境信息——AI 也一样。

⚙️ Harness + Loop：让信息环境自动运转

当 AI 不再是你一句我一句的对话伙伴，而是被编排进自动化循环中自主运转时，信息环境的构建从"每次手动贴信息"升级为"设计一套规则让系统自动获取信息"。

四层架构（层层嵌套、相互托举）

① Prompt 层 —— "怎么问" 角色设定 · 输出格式 · Few-shot 示例 ↓ ② Context 层 —— "让 AI 看到什么" RAG · 记忆管理 · 单事实源 · 上下文压缩 ↓ ③ Harness 层 —— "AI 在什么环境里工作" 工具沙箱 · 权限控制 · 验证闭环 · 子代理 ↓ ④ Loop 层 —— "AI 做完一步后怎么办" 自动发现 → 计划 → 执行 → 验证 → 迭代 → 下一轮

🔑 权力让渡的本质：权力让渡的每一步——从「怎么写」到「信息获取方式」到「执行环境的控制权」到「任务流转的决策权」——都在降低人对 AI 的微观控制，但都在提高人对信息环境设计的要求。未来的核心能力，不是写好 Prompt，而是设计好 AI 赖以运作的信息世界。

六、行动框架

从现在开始，为 AI 构建信息环境的三步走。

🧠 心智转变的三个触发动作

动作 ①

每次派活前花 30 秒补信息

术语 AI 能不能自己搞懂？数据范围能看出来吗？哪些不能动？

动作 ②

接受"AI 会犯错"并建立纠偏机制

犯错不可怕，纠偏成本极低。上次出错的边界这次更新到环境文档。

动作 ③

停止期待 AI"自己悟"

假设 AI 什么都不懂。每次需要它懂什么，就说清楚。写进文档比口头解释更持久。

📋 信息环境构建 Checklist

  • 写一份团队的业务术语表（30 个核心概念）——放在团队共享目录
  • 梳理 3 个最常用的工作流，写下"如果有 AI 帮忙，哪些步骤最容易出错"
  • 建立个人 CLAUDE.md / 工作偏好文档——让 AI 知道你的决策风格
  • 每次给 AI 派活前，问三个问题：数据在哪？边界在哪？验收标准是什么？
  • 建立团队决策边界文档——明确哪些决策 AI 不能自动做
  • 为 3-5 个高频场景编写 Skill（可复用知识编码）
  • 接入 2-3 个关键数据源（数据库、API、文档系统）——建立 Connector
  • 建立"AI 偏差记录"——每次 AI 出错，更新到环境文档
  • 部署 Agent Loop 自动化——配置 Automations 触发源
  • 建立三层飞轮：个人记忆 → 团队知识库 → 组织知识体系
  • 设定信息环境健康度指标：AI 首次成功率、纠偏轮次、指令抽象程度
  • 定期 Review 环境文档质量——就像 Code Review 一样 Review 知识库
