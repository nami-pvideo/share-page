# AI 编码协作的信息分层模型

> 同一个 AI 模型，同样的能力，为什么有时改一行代码一气呵成，有时反而把不相干的模块全改了？核心差别不在于 AI 的智力，而在于它"看到"了什么样的信息结构。本文从信息分层理论出发，系统性拆解 AI 在编码协作中到底缺少什么，以及如何补齐。

---

## 一、原理篇：AI 的信息视野如何构成

### 1.1 三层信息模型

AI 在每次会话中拥有的信息可以清晰地分成三层。理解这三层，是理解 AI 协作行为的基础。

| 层次 | 内容范围 | AI 默认状态 | 典型案例 |
|------|---------|------------|---------|
| **世界知识** | 编程通识、语言语法、主流框架、设计模式、网络协议 | **完整** | Go 的 goroutine 调度、React Hooks 规则、RESTful API 设计 |
| **团队共识** | 项目架构边界、领域术语定义、命名规范、协作约定、历史决策 | **空白** | "订单"和"工单"的区别、为什么 Service 层不能调 Handler 层、灰度策略选型理由 |
| **当前需求** | 本次任务的改动范围、验收标准、上下游影响、限制条件 | **空白** | 这次只改 Provider 层、不要动 Gateway、验收标准是跑通 XXX 测试 |

**第一层世界知识**——AI 在这里是顶级专家。它可以熟练地写各种语言的代码，了解几乎所有主流框架的用法，甚至比许多工程师对标准库的细节更熟。

**第二层团队共识**——AI 在这里是彻头彻尾的"新人"。它不知道你们项目里把"任务"定义成什么，不知道为什么 Gateway 层和 Hook 层必须分开，不知道你们团队约定的代码风格底线。这些信息只存在于团队的 `CONTEXT.md`、`CLAUDE.md`、架构决策记录（ADR）、PRD 和历史口头的讨论里。

**第三层当前需求**——AI 在这里更是完全无知。这次到底要改哪几个文件？验收标准是什么？哪些文件绝对不能动？这些信息只活在工程师的脑子里，或者在即时通讯的对话记录里。

**核心结论**：AI 带着第一层满载而来，但第二层和第三层完全是空的。工程师在协作中的核心角色，就是把第二层和第三层补给它。

### 1.2 从信息论看协作失败

可以把 AI 协作视为一个信息传递系统：

```
信息源 → 编码（指令+文档） → 信道（上下文窗口） → 解码（AI 生成代码） → 输出
```

在这个系统中，故障点通常在两个环节：

1. **编码不足**：工程师想的是"优化一下错误处理"这句在脑子里有完整画面的十个字，但编码给 AI 的也只有这十个字。信息丢失量巨大。
2. **信道过载**：反过来，也有人把整个仓库几万行日志一次性塞进去，期望 AI "自己从中提炼出问题"。这相当于在信道上发送大量噪音，有效信号被淹没。

优秀的 AI 协作，本质上是在做**信息编码与信道优化**——选择性地将团队共识和当前需求以 AI 可消费的形式送入上下文窗口。

### 1.3 业界对这一模型的应用

这个三层模型不是孤例。在 AI 编码工具生态中，多个主流方案都在不同程度上应用了类似思路：

- **Claude Code 的 CLAUDE.md**：自动在每次会话开头注入项目级行为约束，本质上是在把"团队共识"层自动化。
- **Cursor Rules（.cursorrules）**：类似机制，允许团队定义编码规范、目录约定、命名规则，在每次对话中自动生效。
- **GitHub Copilot 的 Workspace Rules**：2024 年新增特性，让团队通过 `.github/copilot-instructions.md` 定义项目级行为准则。
- **Devin 的 Knowledge Base**：Cognition 公司的 Devin 允许配置持久知识库，存储项目架构、API 文档、部署流程等"团队共识"。

这些工具的共同点：它们都意识到光有世界知识级的 AI 不够用，必须给 AI 提供项目级共识。它们的差异在于提供方式——有的自动读取文件，有的手动引用。

## 二、实践篇：如何补齐 AI 的信息缺口

### 2.1 诊断：你的指令停留在哪一层

在派活给 AI 之前，快速自查这三个问题：

| 诊断问题 | 如果答案是"不知道" | 需要什么 |
|---------|-------------------|---------|
| 这个需求里出现的专业术语，一个外部招聘的中级程序员能猜到是什么意思吗？ | AI 也猜不到 | 需要术语定义（团队共识层） |
| 涉及的代码风格、目录结构、注册方式，能从框架官方文档推断出来吗？ | AI 也会按惯例瞎做 | 需要架构约束文档（团队共识层） |
| 这次要改哪几个文件、不能动哪些文件，AI 能在指令里直接看到吗？ | AI 会自由扩散 | 需要明确边界（当前需求层） |

只要有一个问题的答案是"不知道"，那就意味着 AI 缺少某一层的关键信息，需要在指令或文档中补充。

### 2.2 业界实践：三类补充方案

不同场景下补信息的方案不同。业界常见的有三类：

**方案一：项目级持久文档（补齐第二层）**

这是最基础也最持久的方案。代表实践：

- **CLAUDE.md / AGENTS.md**：项目根目录的 Markdown 文件，定义架构分层、目录约定、命令规范、禁止事项。Claude Code、Codex、Gemini CLI 等工具会在每次会话开头自动读取。
- **CONTEXT.md**：领域术语表。每条术语包含名称、定义、与其他概念的关系。消除"AI 把任务和作品混着写"这类歧义错误。
- **Architecture Decision Records（ADR）**：记录关键架构决策及其理由。让 AI 不仅知道"怎么做的"，更知道"为什么这么做"。

这条方案适合**长期稳定的团队共识**——架构边界、术语定义、核心规范。变更频率低，但每次变更都需要同步更新文档。

**方案二：需求级一次性指令（补齐第三层）**

这是最灵活但最容易被省略的方案。代表实践是一个三段式指令结构：

```text
【目标】
一句话描述最终要达成的事。

【起点】
- 核心参考文件：xxx/xxx.go（告诉它从哪开始看）
- 相关领域术语见 CONTEXT.md 的 XX 段（解除歧义）
- 本次需求文档见 docs/prd/xxx.md（给范围）

【边界】
- 要改的文件：a.go, b.go
- 不要动的文件/层/模块：Gateway 层、Hook 层
- 超出范围的发现：只提出来，不要直接改
```

这段指令做的不多，但它把 AI 的探索半径从"整个仓库"框到了"几个文件"。

**方案三：运行时按需注入（动态补齐）**

适用于可能用到但不一定用到的辅助信息。代表工具是 **MCP（Model Context Protocol）**——一个由 Anthropic 推出的开放协议，让 AI 在需要时通过标准化接口获取知识库、数据库、API 文档等内容。

典型场景：
- AI 需要查某个业务接口的详细字段定义 → 调用 MCP 拉取 API 文档
- AI 需要知道某个配置项的合法值范围 → 调用 MCP 查询配置中心
- AI 需要查看历史决策背景 → 调用 MCP 搜索 ADR 存档

Anthropic 的 MCP 已被 Claude Desktop、Claude Code 支持，Cursor 和 Continue 也已集成类似机制。

### 2.3 正反面对比

**场景一（信息完整）**：

指令中包含明确的参考实现路径。"参考 `users/service.go` 的实现风格，给 `orders/service.go` 加一个批量查询方法，签名和错误处理跟参考文件保持一致，只动这一个文件，不要改 import 部分以外的任何代码。"

AI 输出：改了一个文件的一个方法，签名对、错误处理一致、import 干净，一次通过。

**场景二（信息缺失）**：

指令只有一句："给订单模块加个批量查询。"

AI 动作：
1. 自主 grep "order" 关键词，触达到 12 个文件
2. 把 OrderService 和 OrderRepository 都改了
3. "顺手"统一了错误处理风格，改了 4 个无关文件
4. 给的解释很漂亮："为了让批量查询与现有逻辑保持一致的错误传播路径"

根因：AI 在没有边界信息的情况下，默认"一切相关"；在缺少风格参照的情况下，默认"按自己理解统一"。

**信息分层模型的工程价值**：不是让 AI 更聪明，而是让 AI 不需要猜。

## 三、总结

AI 与工程师协作的核心约束是**信息流转**。AI 不会读心术，每次会话都从零开始——它只有世界知识，没有团队共识和当前需求。

补齐信息缺口的三条主线：

| 方案 | 补齐哪一层 | 载体 | 时机 |
|------|----------|------|------|
| 项目级持久文档 | 团队共识 | CLAUDE.md / CONTEXT.md / ADR | 项目初期建立，持续维护 |
| 需求级一次性指令 | 当前需求 | 三段式指令（目标+起点+边界） | 每次派任务 |
| 运行时按需注入 | 两者均可 | MCP 等工具接口 | AI 主动拉取 |

记住一件事：AI 每少知道一点，它就会拿"猜测"补上——而它的猜测，往往就是工程师 review 时头疼的来源。
