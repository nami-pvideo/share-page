# 信息披露管理：提升 AI 编程效率的核心杠杆

> 同一个 AI 模型，同样的能力，为什么有时改一行代码一气呵成，有时反而把不相干的模块全改了？核心差别不在于 AI 的智力，而在于它"看到"了什么样的信息结构。

---

## 1. 信息分层模型 —— AI 的"已知"与"未知"

AI 在每次会话中拥有的信息，可以清晰地分成三层。理解这三层，是理解 AI 协作行为的基础。

**第一层：🌍 世界知识（AI 默认为满）**

编程通识、语言语法、主流框架、设计模式、网络协议 —— AI 在这里是顶级专家。Go 的 goroutine 调度、React Hooks 规则、RESTful API 设计，随问随答。

**第二层：🏢 团队共识（AI 完全空白）**

项目架构边界、领域术语定义、命名规范、协作约定、历史决策。AI 不知道"订单"和"工单"的区别，不知道为什么 Service 不能调 Handler。

**第三层：📋 当前需求（AI 完全空白）**

本次任务的改动范围、验收标准、上下游影响、限制条件。AI 不知道这次只改 Provider 层，不知道验收标准是跑通 XXX 测试。

信息从"工程师的脑子"到"AI 的输出"经历了一条长链路：信息源 → 编码（指令+文档）→ 信道（上下文窗口）→ 解码（AI 分析）→ 输出（代码）。串联这条链路中间，信息丢失量巨大，有效信号极易被淹没。**优秀的 AI 协作 = 信息编码优化 + 信道管理**——选择性将团队共识和当前需求以 AI 可消费的形式送入上下文窗口。

> **💡 核心认知：**AI 带着第一层满载而来，但第二层和第三层完全是空的。工程师在协作中的核心角色，就是把第二层和第三层补给它。

## 2. 信息披露的本质 —— 从抽象到具体的距离

信息从"一个人的脑子里"变成"团队和 AI 都能读取的文档"，经历了一个由模糊到清晰的四层光谱：

```
潜意识 → 个人模糊掌握 → 个人充分掌握 → 团队共识
```

绝大多数 AI 翻车，根因是需求停留在前两层就甩给 AI 了。

### "狗"与"你家那只金毛"——抽象与具体的距离

| 抽象级别 | 类比 | 编程中的对应 | AI 能交付吗？ |
| --- | --- | --- | --- |
| 品种泛指 | "帮我带只狗回来" | "改一下这个页面" | ❌ 会交付一只"随机狗" |
| 品种具体 | "带一只金毛回来" | "把支付按钮置灰" | ⚠️ 方向对了但细节不对 |
| 个体具体 | "带邻居家叫 Lucky 的金毛" | "当余额不足时，支付按钮 disabled，token=--btn-disabled，hover 提示'余额不足'" | ✅ 交付你想要的 |

### 最小信息披露原则

不是"把所有信息都告诉 AI"，而是"只告诉 AI 它不能自己推断出来的信息"。

- **✅ AI 自己能推断 → 不需要披露**：Go 标准库用法、主流框架常见用法、通用设计模式、编程语言语法
- **❌ AI 推断不出来 → 必须披露**：项目特定的命名约定、架构约束、术语定义、本次需求的范围和限制

> **💡 一句话：**信息说不清，AI 替你猜；猜对了是你运气好，猜错了是你 review 的工时。下次给 AI 派活前，问自己一句——它在什么地方需要猜？

## 3. 上下文管理 —— AI 的"工作内存"

上下文窗口不是"能塞多少塞多少"的免费资源，而是要精打细算的稀缺品。

### 四种信息投喂策略

- **📥 主动塞（Push）**：对话开始时塞入核心上下文（PRD 核心段落 + 2-3 个主链路文件 + 边界），锁定方向。**注意：**只塞 2-3 个核心文件，塞多了反而稀释注意力。
- **🔍 被动探索（Let Agent Search）**：只说目标，让 Agent 自己去 grep、读文件、跑命令。适合问题定位不明确的开放性问题。
- **🔗 按需取（Pull on Demand）**：通过 MCP 等接口，让 AI 在需要时自己去拉取信息，不预占上下文窗口，按需"付费"。
- **⚡ 混合策略**：前期 Push 定准方向 → 中期 Agent 在安全半径内自主推进 → 后期按需补细节。全程由 CLAUDE.md / AGENTS.md 自动注入团队共识层。

## 4. Agent vs Workflow —— 两种 AI 模式的信息观

Agent 和 Workflow 最本质的区别，不在于"谁更智能"，而在于**对待信息缺失的态度**——一个会主动去找，一个必须提前给全。

### 核心差异：信息缺失时，谁兜底？

**🤖 Agent 模式："信息不够？我自己去找。"**

Agent 的设计哲学是**容忍信息缺口**。你只需要给一个方向，缺失的信息由 AI 自主补充——它通过**工具调用、文件检索、反问澄清、文档注入**四条路径去拉取未知信息。

**⚙️ Workflow 模式："信息不全？直接报错。"**

Workflow 的设计哲学是**零容忍信息缺口**。每一步的输入必须在编排阶段就定义清楚——缺少任何一个参数，步骤直接中断，没有"自己去补"这回事。

### 信息缺失的两种后果

| 场景 | Agent 模式的反应 | Workflow 模式的反应 |
| --- | --- | --- |
| 工程师只说了"改一下订单模块" | 自动 grep "order"，找到相关文件，读代码理解结构，再定位改动点 | 报错：缺少"要改哪个文件"的输入参数 |
| 不知道配置项的合法值范围 | 调用 MCP 查询配置中心，或反问工程师 | 报错：配置校验失败 |
| 发现"订单"和"工单"两个概念可能混淆 | 读 CONTEXT.md 确认定义，或反问确认 | 不会发现——按预设流程盲走，交付错误结果 |
| 需要了解某个历史决策的背景 | 搜索 ADR 存档，或读相关 PRD 文档 | 不会尝试——只有预定义的输入 |

> **💡 关键洞察：**Agent 的"自主信息拉取"能力，意味着**信息披露不需要完美**——你不必在事前把一切可能用到的信息都准备好。Agent 把"信息缺口"从**阻塞点**变成**自动处理的中间步骤**。

### 从信息披露角度看选型

- **选 Agent**：你无法在事前穷举所有信息 / 需要跨文件跨模块探索 / 需求可能在中途出现意外分支 / 任务是一次性的
- **选 Workflow**：输入输出事前完全确定 / 不需要"探索"或"判断" / 需要定期无人值守执行 / 对稳定性要求高于灵活性

> **💡 选型口诀：**信息能穷举的用 Workflow，信息有缺口的用 Agent。日常开发 95%+ 的需求天然带有信息缺口——这正是 Agent 的用武之地。

### 让 Agent 高效获取信息的四大工程支撑

1. **🛡️ Harness 安全围栏**：给 Agent 装"安全围栏"——能读哪些文件、能跑哪些命令、改代码前是否需要确认。信息获取在可控范围内。
2. **📂 Git Worktree 隔离空间**：每个 Agent 独立工作目录，互不干扰。Agent 可以放心探索，不会污染其他人的工作区。
3. **📋 Skill 系统 SOP**：brainstorming → plans → execute → verify。告诉 Agent 什么时候该反问、什么时候该拉取、什么时候该验证。
4. **📚 文档中心 长期记忆**：CLAUDE.md + CONTEXT.md + ADR + PRD。Agent 的信息获取不是从零开始——持久文档是它的"已知信息基线"。

## 5. 新员工心智模型 —— 像带新人一样带 AI

差别往往不在技术层面，而在一个基础认知：你到底是把 AI 当"全知全能的代码生成器"，还是当"能力很强但什么都不懂的第一天入职员工"？

### 类比映射表

| 新员工入职需要什么 | 映射到 AI 协作 | 工程实现 |
| --- | --- | --- |
| 入职资料包（项目简介、术语、规范） | 补齐团队共识层 | CLAUDE.md + CONTEXT.md + ADR |
| 一张干净的工位和电脑 | 独立的执行环境 | Git Worktree + 独立会话 |
| 师傅告诉 TA"从哪开始、哪些不能动" | 三段式指令 | 每次派活时的明确指令 |
| 师傅的工位旁边可以随时问 | 对话式交互 | Agent 模式的思考-行动循环 |
| 公司的工作 SOP | 标准化 AI 协作流程 | Skill 系统（四步流程） |
| 犯错后师傅指正 | Review + 反馈 + 更新约束文档 | Code Review → 写进 CLAUDE.md |

### 三段式指令模板

好的 AI 任务指令，跟带新员工时的交代几乎一样：

```
【目标】
给订单模块的 CreateOrder 函数加一个并发锁，防止超卖。

【起点】
- 核心参考文件：orders/service.go 的 CreateOrder 函数（第 45 行）
- 参考实现：inventory/service.go 里的锁实现（完整、线上验证过的写法）
- 领域术语：见 CONTEXT.md 的"订单 → 库存扣减"段

【边界】
- 要改的文件：orders/service.go（只改这一个）
- 不要动的层：Gateway 层、Handler 层、Hook 层
- 不要改 import 部分以外的任何代码
- 超出范围的问题：只提出来，不要直接改

【验收】
跑 go test ./orders/... -run TestCreateOrder -count=10，全部通过即为完成。
```

> **💡 心智转变的三个触发动作：**
> ① 每次派活前花 30 秒补信息——术语 AI 能不能自己搞懂？文件范围能看出来吗？哪些不能动？
> ② 接受"AI 会犯错"并建立纠偏机制——犯错不可怕，纠偏成本极低（回滚即修复）
> ③ 停止期待 AI"自己悟"——假设 AI 什么都不懂，每次需要它懂什么，就说清楚

## 6. 实战 Checklist —— 上手就能用

不需要一次全部做到。从最下面的一条开始，每一周加一条，一个月后整个协作模式会发生质变。

**第 1 周：建立基础文档**

- 在项目根目录创建 `AGENTS.md`，写清：架构分层（哪层能调哪层）、目录约定、禁止事项
- 创建 `CONTEXT.md`，定义 3-5 个最容易被 AI 搞混的核心术语
- 确保 AI 工具配置了自动读取 AGENTS.md

**第 2 周：规范指令格式**

- 每次派活用三段式指令：目标 + 起点 + 边界
- 明确告诉 AI 哪些文件/模块不能动
- 给出验收标准——跑什么命令、哪个测试通过就算完成

**第 3 周：引入 Worktree 并行**

- 安装 git worktree 工作流（`git worktree add ../proj-feat feat/xxx`）
- 每个 AI 任务开独立 worktree + 独立会话
- 不同任务不混在同一个会话里

**第 4 周：建立纠偏闭环**

- Review AI 代码时，发现系统性问题马上写进 AGENTS.md（如"Provider 改动不触及 Gateway"）
- 每次 AI 犯错 = 一次文档升级的机会
- 把常用模式沉淀为 Skill（SOP），让 AI 不跳关键步骤

## 7. 反模式速查 —— 常见的坑与修正方案

| 反模式 | 修正方案 |
| --- | --- |
| ❌ 一次性塞入大量日志（几千行只有 2-3 行关键错误） | ✅ 先用 shell 过滤 ERROR/FATAL，只送有价值上下文 |
| ❌ 把整个 README 贴进对话（AI 完全可以自己读） | ✅ 让 AI 自己读文件，不要主动贴 |
| ❌ 一个会话混做多个任务（旧任务信息占着上下文） | ✅ 不同任务开不同会话，一条会话一条事 |
| ❌ 主动塞入"可能用到"的所有文件（AI 在海量信息中迷失） | ✅ 只贴 2-3 个核心文件，AI 需要更多自己读 |
| ❌ 上下文快满时追加关键信息（可能被压缩工具"挤掉"） | ✅ 开新会话，总结当前结论作为起点 |
| ❌ 超长 Prompt 模板（几百行模板吃掉一小半窗口） | ✅ 长期规则放 CLAUDE.md，对话中只写本次直接相关的 |

## 8. Harness 与 Loop —— 自动化场景下的信息披露

当 AI 不再是你一句我一句的对话伙伴，而是被编排进自动化循环（Loop）中自主运转时，信息披露面临一个全新的挑战：**没有人站在旁边随时纠正**。Harness 和 Loop 如何协作，让 Agent 在无人值守时依然获得足够且正确的信息？

### 8.1 从对话到自动化：信息披露的范式跃迁

AI 工程化经历了三次进化，它们不是替代关系，而是**层层嵌套、相互托举的共生体**：

```
① Prompt 层 —— "怎么问"：角色设定 · 输出格式 · Few-shot 示例
    解决：语义对齐 · 推理引导 · 信息编码的第一步
    ↓
② Context 层 —— "让 AI 看到什么"：RAG · 记忆管理 · 单事实源 · 上下文压缩
    解决：信息检索 · Token 分配 · 跨会话记忆 · 上下文窗口优化
    ↓
③ Harness 层 —— "AI 在什么环境里工作"：工具沙箱 · 权限控制 · 验证闭环 · 子代理
    解决：边界约束 · 工具编排 · 反馈闭环 · 安全围栏
    ↓
④ Loop 层 —— "AI 做完一步后怎么办"：自动发现 → 分发 → 检查 → 记录 → 下一轮
    解决：自动化运转 · 闭环验证 · 无人值守 · 信息披露从手动变为规则驱动
```

前三层的职责边界：**Prompt** 解决"如何对模型说清楚指令"，**Context** 解决"模型需要知道哪些背景"，**Harness** 解决"整个 AI 系统如何高可靠、可控地运转"。而 **Loop** 则把三者的信息流串成一个自驱动闭环——人不再逐条给信息，而是**事先定义信息获取规则**，Loop 在每一轮自动执行这些规则。

### 8.2 Loop 五阶段循环：每个阶段的信息披露节点

每一个 AI 编码循环——无论是单 Agent 还是多 Agent——都遵循相同的五阶段循环，直到满足可验证的停止条件。每个阶段都有**特定的信息披露关键动作**：

```
🔍 ① Discover（发现）
    信息披露动作：从外部状态读取"什么需要做"
    信息来源：CI 失败日志 · Issue 系统 · Git diff · 定时任务输出
    ↓
📋 ② Plan（计划）
    信息披露动作：拉取项目规范 + 历史决策确定"怎么做"
    信息来源：CLAUDE.md · Skills · ADR · CONTEXT.md
    ↓
⚡ ③ Execute（执行）
    信息披露动作：在隔离 worktree 中精确获取目标代码
    机制：子代理 · 独立上下文窗口 · 工具白名单
    ↓
✅ ④ Verify（验证）
    信息披露动作：用不同模型做独立检查，杜绝"自评自"
    机制：独立的 Evaluator 子代理 · 高 reasoning 模型 · 不共享上下文
    ↓
🔁 ⑤ Iterate（迭代）
    信息披露动作：写入 STATE.md · 更新 Issue · 记录哪些成了哪些没成
    核心原则：模型会遗忘，仓库不会——所有状态存在外部系统
    ↓ (回到 ①)
```

> **💡 Loop 的核心哲学：**不要信任模型的上下文窗口作为持久化存储。模型会遗忘、会漂移、压缩信息时会导致约束丢失。每个循环迭代都从一个**全新的上下文窗口**开始，但需要在实际持久化的内容（STATE.md、Issue 系统、Git 仓库）基础上工作。这就是为什么最原始的 Ralph Loop 只用了一行 `while :; do cat PROMPT.md | claude-code; done` 就实现了永续自动化——每次循环重新读取 PROMPT.md 和当前代码库状态，完全忽略之前的对话历史，反而解决了所有长对话上下文带来的问题。

### 8.3 Harness：自动化场景的信息披露护栏

在 Loop 驱动的自动化场景中，Harness 承担了"确保 Agent 拿到正确信息、不会拿错信息、不乱用信息"的职责。Claude Code、Hermes Agent 和 OpenClaw 三条路线，对"如何在 Loop 中管理信息披露"给出了三种不同的答案。

**🔒 Claude Code：信任模型 + 管好边界**

信息披露哲学：**模型自己能搞定的事就别替它做**。Harness 不替模型做决策，只在两个地方卡住：有没有权限做这件事、上下文是不是快溢出了。

- 信息获取：CLAUDE.md 文件前缀加载 + Auto Memory（模型自己写 Markdown）
- 权限管控：六种权限模式 · 只读工具自动放行 · 副作用工具每次检查
- 上下文保护：反应式压缩 · fork 独立实例做摘要 · 3 次失败 circuit breaker
- Loop 中的信息流：每轮完整历史交给模型 · 工具执行结果流式返回

**📚 Hermes：分层信息积累**

信息披露哲学：**光给模型看到不够，得帮它在多个时间尺度上积累和召回**。

- 五层记忆：MEMORY.md → SQLite+FTS5 全文搜索 → 8 个外部 memory provider
- 渐进披露：Skills 分三级加载（名称→内容→附件），模型先扫描再深入
- 程序性记忆：Agent 完成任务后把方法保存为技能文件，下次复用
- Loop 中的信息流：每轮前预取相关记忆 → 每轮后同步对话 → 会话结束抽取长期记忆

**🔌 OpenClaw：信息获取作为可插拔服务**

信息披露哲学：**信息管理的需求会变，与其内置固定方案，不如把接口留好**。

- Context Engine：可替换运行时插件，四个生命周期点（ingest / assemble / compact / after turn）
- 会话隔离：DM 共享 session · 群聊按 group 隔离 · cron 每次新 session
- 多 Agent：每个 Agent 独立 workspace + auth profile + session store
- Loop 中的信息流：Gateway 统一协议 → 所有入口收敛到同一套会话管理和工具策略

> **💡 "Harness 衰变定律"：**Anthropic 的核心观察——**模型原生能力 ∝ 1 / 所需 Harness 的复杂度**。模型越强，需要的 Harness 越薄。这意味着信息披露设计应该**为减法而设计**：每个 Harness 组件都应该能被安全移除，而不是越堆越厚。模型能自主获取的信息，就不要写死在 Harness 里。把 Harness 收拢在两类长效场景中：① 绝对的合规与业务硬边界（金融风控、权限隔离、Human-in-the-loop）；② 外部环境的互操作系统（API 密钥、数据库连接、状态存储）。

### 8.4 Loop 六大核心要素的信息披露设计

所有现代 AI 编码工具（Claude Code、Codex 等）都内置了 Loop Engineering 所需的六大核心要素。从信息披露的角度重新审视它们：

| 要素 | 在信息披露中的角色 | 典型实践 |
| --- | --- | --- |
| ⏰ **Automations**（自动化） | **信息触发源**：定义"什么时候该去发现问题"。没有 Automation，信息就躺在那里，Agent 不会主动去看。 | 定时运行 CI 失败分类、每日 issue 审查。Claude Code: `/loop` 定时 + `/goal` 满足条件才停 |
| 🌲 **Worktrees**（工作树） | **信息隔离空间**：每个 Agent 看到的是独立副本，互不污染。杜绝"A 改了 B 以为没改"的信息不一致。 | 每个任务独立 worktree + 独立分支，完成后清理 |
| 📦 **Skills**（技能） | **信息复用机制**：把"每次都要重新解释"的项目知识编码到磁盘，Agent 按需读取。告别"intent debt"。 | SKILL.md + 可选脚本 + 参考资料。Agent 发现任务匹配时自动调用 |
| 🔗 **Connectors**（连接器） | **信息管道**：让 Loop 触及文件系统之外的信息源——Issue 系统、CI/CD、Slack。没有它 Loop 只看代码，看不到"世界"。 | 基于 MCP 协议接入真实工具 |
| 🤖 **Sub-agents**（子代理） | **信息视角分离**：实现者只拿到任务文件，验证者拿到完整 spec + 验收标准 + 安全规则。对抗"自评自"。 | Explore（用 Haiku）→ Implement（全功能）→ Verify（强模型 + 高 reasoning） |
| 📝 **State**（状态） | **信息持久化的脊梁**：所有状态存在外部，不在上下文窗口里。模型遗忘一切，但 STATE.md 记住一切。 | STATE.md · GitHub Issues · SQLite。记录：已完成 / 进行中 / 尝试过但失败了 / 待处理 |

> **💡 Skills 的信息披露经济学：**Agent 每轮启动都是"冷启动"——它会用自信的猜测填补任何信息缺口。一个 Skill 就是把项目约定、编码规范、历史教训**一次性写在外面**，Agent 每次运行都读取，不再重新推导。没有 Skills，Loop 每轮循环都在从零重新推导项目知识；有了 Skills，信息在循环之间**复利增长**。Hermes 甚至进一步——Agent 完成任务后可以把方法保存为新 Skill，积累 20+ 自创技能后，同类任务速度提升 40%。

### 8.5 闭环 vs 开环：信息披露的安全边界

**🔒 闭环（Closed Loop）：有明确停止条件，Agent 在有限信息范围内运作。**

信息披露五要素：
- **明确的目标（Goal）**：精确定义"完成"的样子
- **充足的上下文**：VISION.md + ARCHITECTURE.md + RULES.md
- **受限的动作**：只暴露必要工具，不给多余权限
- **客观的反馈**：测试、lint、类型检查的自动结果
- **清晰的停止条件**：可验证的成功标准

适合：CI 修复、代码审查、测试生成等边界明确的任务。

**🔓 开环（Open Loop）：持续运行，没有预设终点。需要人类介入来控制信息披露边界。**

信息披露五要素：
- **人类审查机制（Human-in-the-loop）**：关键节点必须人类确认
- **自动分类 TRIAGE**：完成/需审查/放弃——区分不同信息可靠度
- **成本上限硬约束**：max-iterations + 预算封顶
- **优先级排序**：从问题严重性决定"先看哪个信息"
- **异常检测**：如果输出质量偏离基线，自动降级到闭环

> ⚠️ 开环风险极高，中等代码库 50-100 次迭代可烧 ¥500-1000。**建议从闭环开始。**

**🔌 熔断机制（Circuit Breaker）—— 开环的最后一道信息护栏**

无人值守的 Loop 必须有硬性停止条件，否则一个陷入"反复尝试同一个错误修复"的 Agent 会悄悄烧光预算。生产级参数参考：

- **最大迭代**：15–25 步
- **单轮超时**：~300 秒
- **单次成本预算**：~$2

实用技巧——**对每轮迭代的动作取哈希指纹，连续 3 次相同立即停止**（指纹重复说明 Agent 已陷入死循环，再喂信息也无济于事）。从信息披露视角看，这是一条"负向边界"：当 Agent 不再产生**新信息**时，及时切断比继续供给信息更重要。

> Anthropic 研究员有句话点破本质：*"Rubric design is the skill now, the model is the easy part."*（设计可验证的成功/停止标准，才是今天真正的工程能力。）

### 8.6 实战：一个 CI 自动修复 Loop 的信息披露设计

以"每天早上自动修复前一天 CI 失败"为例，拆解每个环节的信息披露设计：

```
⏰ 6:00 AM 触发（GitHub Actions 定时 / ci-fix-loop.yml）
  → 信息披露：定义触发源
    ↓
🔍 读取 CI 失败日志（ci-triage Skill 分析，分类：可自动修复 / 需人工）
  → 信息披露：信息源筛选——只看 CI 失败日志，不看完整的代码库
    ↓                              ↓（复杂问题，无法自动修复）
🤖 子代理修复 + 审查                    👤 需人工审查（人机边界）
  ci-fixer（实现）→ code-reviewer     → 信息披露：标记为需人工介入
  （验证，独立 worktree · 不同模型 · 
   隔离上下文）
  → 信息披露：信息视角分离
    ↓
📊 写入状态 + 打开 PR（STATE.md · GitHub PR · Issue 更新）
  → 信息披露：持久化 + 人类可审
    ↓
🔁 次日 6:00（读取 STATE.md，从上次中断处继续）
```

这个 Loop 的信息披露设计核心：**每个环节都知道"该从哪里取信息、该把信息写到哪里"——不是靠记忆，而是靠外部化规则。**

### 8.7 Harness + Loop 信息披露六大设计原则

1. **📌 状态必须外部化**：模型会遗忘，上下文会压缩。所有 Loop 状态写入 STATE.md 或 Issue 系统，不在上下文窗口靠记忆。
2. **📌 信息视角必须分离**：实现者看到"怎么做"的信息，验证者看到"对不对"的信息。用 Sub-agent + 不同模型 + 独立上下文强制执行。
3. **📌 Skills 是信息的一次性编码**：不要在每个 Loop 迭代中重新解释项目约定。把约定写成 Skill，让 Agent 按需加载——信息在迭代间**复利增长**。
4. **📌 Harness 为减法而设计**：模型在进步，今天需要的硬编码约束明天可能是死代码。只把"绝对的合规硬边界"和"外部环境连接器"写死在 Harness 里。
5. **📌 从闭环开始，谨慎开环**：闭环有明确信息边界——Agent 知道自己该看什么。开环的信息获取是无限度的，必须有成本上限、迭代限制和人工审查节点。
6. **📌 Connectors 让 Loop 看到"世界"**：没有 MCP Connectors，Loop 只能看到代码文件系统。接入 Issue 系统、CI/CD、Slack，Loop 才能实现"发现问题→修复→PR→通知"全闭环。

> **💡 终局洞察：人类掌舵，Agent 执行。**Loop + Harness 的信息披露设计，本质上就是把"人类每次给 AI 补信息"这件事，从**手动操作**变成**系统规则**。你的角色从"每次对话的信息搬运工"升级为**"信息获取规则的设计者"**——这是一次根本性的工作方式升维。
>
> 正如 Claude Code 负责人所说：*"I don't prompt Claude anymore. I have loops running that prompt Claude and figuring out what to do. My job is to write loops."*
