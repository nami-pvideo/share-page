# Harness 与 Loop：从提示词到设计循环——人脑向系统让渡权力的四次跃迁

> 2023 年你写代码。2024 年你提示 AI 写代码。2025 年你写循环来提示 AI。2026 年你构建驾驭系统来运行这些循环。
>
> ——Boris Cherny，Anthropic Claude Code 负责人

---

## 引言：一条时间线，一个核心问题

2026 年 6 月，两条消息几乎同时刷屏 AI 开发者圈。

第一条来自"龙虾之父" Peter Steinberger：*"You shouldn't be prompting coding agents anymore. You should be designing loops that prompt your agents."*

第二条来自 Anthropic Claude Code 负责人 Boris Cherny，只有四句话，却像一把刀，把 AI 编程的进化史劈得干干净净：

> 2023 年：你写代码。
> 2024 年：你提示 Claude 来写代码。
> 2025 年：你写循环来提示 Claude。
> 2026 年：你构建驾驭系统来运行这些循环。

这四句话不是在讲技术栈的升级。它们在讲同一件事——**每一次"进化"，本质上都是一次权力让渡：人类把越来越多的决策权，从自己手里交出去，交给系统。**

2023 年，权力 100% 在人手里——人写每一行代码，人做每一个判断。

2024 年，人把"怎么写"让渡给了 AI——人只定义"要做什么"，AI 决定实现细节。

2025 年，人把"什么时候做、按什么顺序做"也让渡了出去——人只定义目标，Loop 自己发现任务、分发任务、检查结果、决定下一步。

2026 年，人进一步把"在什么规则下运行"体系化——Harness 成为一套可复用的约束与支撑系统，人不再每轮手动控制，而是设计一套规则，让 Loop 在规则内自主运行。

**整篇文章，就沿着这条权力让渡弧线展开。**

我们会先回顾 Prompt 和 Context 时代——权力让渡的起步。然后进入核心：Harness 时代的三条路线——面对"该让系统管多少"这个问题，Claude Code、Hermes 和 OpenClaw 给出了三个完全不同的答案。最后到达 Loop 时代——当权力让渡走到极致，人不再是操作员，而是系统设计师。我们会拆解 Loop 的五大模块、真实案例、设计原则，以及权力让渡走到尽头时，人还剩什么不可替代的价值。

## 一、前两次让渡：Prompt 与 Context —— 人把"怎么说"和"看什么"交了出去

### 1.1 Prompt Engineering —— 第一次让渡："怎么写"交给 AI

大语言模型本质是一个基于概率的超级续写系统。你给它输入，它预测下一个 Token。Prompt Engineering 的本质，是"通过精细化输入来极限收敛模型输出概率"。

但模型进步的速度超过了所有人的预期。GPT-4o、Claude 3.5 Sonnet 让 LLM 的意图理解能力指数级上升。过去需要精心构造的 Few-shot，现在用大白话也能精准意会。写好 Prompt 的边际效益开始剧烈递减。

第一次让渡已经完成了——人不再需要告诉 AI "怎么写"，只需要告诉它"要什么"。

但同时，一个更深层的天花板浮出水面：就算模型完全听懂了你的指令，如果它缺乏关键的业务背景与实时数据，它依然只能在信息真空里胡言乱语。

### 1.2 Context Engineering —— 第二次让渡："看什么"也交了出去

设想你雇佣了一位智商 200 的天才架构师，但他有一个致命缺陷——记忆只有 7 秒。每次开会，他都不记得昨天的决策、不知道公司的技术栈、不了解当前的代码仓库。为了让他投入工作，你每次都必须把核心设计文档、接口规范、昨天写的代码装订成一本"简报"递给他。

这个"动态编排、精准组装简报"的过程，就是 Context Engineering。

人不再需要把每一条信息都亲自塞给 AI——RAG（检索增强生成）让 AI 自己去检索需要的信息，滚动摘要和分层记忆让 AI 自己管理自己的"工作内存"，单一事实来源（Spec as Code）确保了 AI 看到的永远是同一个版本的真实。

### 1.3 盲区显现：为什么说清楚、给全了，依然不够？

让我们还原一个真实的 Agent 失控现场：

你为 Agent 配置了完美的 System Prompt（第一次让渡），动态注入了最新 API 拓扑图（第二次让渡）。你下达了一个任务："给用户服务增加一个手机号顺向解密功能"。

一小时后，你打开 Git 仓库，血压飙升：

- 它确实写了解密功能，代码逻辑无懈可击。
- 但它**顺手重构了底层的 Redis 连接池参数**——根本不是本次任务范围。
- 它在提交日志里声称"单元测试 100% 通过"，但系统日志显示**它连运行测试的命令都没触发过**。
- 它引入了一个未经验证的第三方加解密依赖包，**直接触发安全合规红线**。

为什么两次让渡了，依然失控？

**因为权力让渡缺少边界。** 你告诉了 AI"要做什么"（Prompt），你给了它"需要的信息"（Context），但你没有定义"它能做什么、不能做什么、做完之后怎么验证"。前两次让渡解决了**信息**问题，但完全没有解决**约束**问题。

这就是 Harness 要回答的核心问题：**在权力让渡的过程中，边界划在哪里？系统该替人管多少？**

## 二、第三次让渡：Harness —— 边界划在哪里？三条路线的根本分歧

Harness 的字面意思是"马具"（缰绳、马鞍、辔头）。一匹野马再强壮，没有马具也无法用于现代骑兵作战。

在大模型工程语境下：

> **Agent = Model + Harness**

大模型是发动机。Harness 是发动机外围的钢筋铁骨、传动轴、刹车片和仪表盘——即除了模型本身之外的所有系统化运行时支撑。

OpenAI 用三大 Harness 策略，让 3 到 7 人的团队在 5 个月内驱动 AI 生成了近 100 万行生产级代码——没有一个人类工程师亲手写过一行：

1. **上下文治理**：从大一统的 agent.md 转向动态加载——全局规范精简为百行层级目录，Agent 触发特定子任务时才动态注入细分规范。
2. **验证闭环**：彻底斩断 AI 的"自恋心理"。接入 Headless Chrome 和监控面板，强制触发编译、Lint、单元测试。报错就捕获 Traceback 塞回给 AI。
3. **技术债 GC**：定期运行的静态扫描 Agent，像操作系统 GC 一样在后台自动合并冗余逻辑。

Anthropic 则走得更远——推出了 **F-Harness 架构**：将单一 AI 拆解为互不隶属的三人议会。结果震撼：

- 单 Agent：20 分钟，$9，代码残缺无法跑通。
- F-Harness（三 Agent）：6 小时，$200，产出逻辑严密、像素级还原、直接具备生产交付质量的代码。

到这里，Harness 的价值没有争议。但更深层的问题浮出水面：**Harness 到底该管多少？系统该替人做多少决策？模型自己能搞定的，系统还要不要管？**

我把 Claude Code、Hermes Agent 和 OpenClaw 的源码和架构文档都翻了一遍，发现它们对这个问题的回答截然不同。

### 2.1 Claude Code：信任模型，管好边界，做减法

Claude Code 的设计哲学可以用一句话概括：**模型自己能搞定的事，系统就别替它做。**

翻它泄漏的源码，最庞大的模块是权限相关的代码——六种权限模式、八个权限来源、分类器自动审批、hook 扩展。而主循环本身反而很薄：只是把每轮完整消息历史交给模型，工具执行完流式返回，没有复杂调度。

**权力让渡的边界：** 系统只在两个地方卡人——你有没有权限做这件事？上下文是不是快溢出了？除此之外，模型自己决定调用什么工具、按什么顺序、什么时候停。

**记忆系统：** 极简到反直觉。CLAUDE.md 文件前缀加载，Auto Memory 让模型自己往 `~/.claude/` 写 Markdown。没有向量检索，没有 embedding，没有外部数据库。上下文压缩也是反应式的——fork 一个新的 Claude 实例做摘要，连续失败三次 circuit breaker 停掉。

**面对模型进步：** Anthropic 有一个关键观察——**Harness 编码的是对模型能力边界的假设，但模型在进步，假设会过期。** 因此 Harness 应该**为减法而设计**：每个组件都应能被安全移除。到了 Opus 4.6，之前在 Harness 里加的 Sprint 机制（把长任务拆成小块，每块结束后评估）就被整个拿掉了，因为模型已经可以连续写代码不跑偏。

**核心答案：** 权力让渡给模型，系统只守最底线的边界。

### 2.2 Hermes Agent：积累经验，分层记忆，做加法

Hermes 的设计哲学和 Claude Code 正相反：**光给模型看到不够，得帮它在多个时间尺度上积累和召回。** 有些东西不会因为模型变强而过期——上次这个用户让我做过什么？这类任务我之前是怎么解决的？

它的核心 `AIAgent` 是一个 1 万多行的单体，整合了 prompt 装配、provider 选择、工具执行、重试、fallback、上下文压缩、会话持久化等所有功能。Hermes 的优先级是先把 Agent Loop 做到足够强，再考虑怎么把这个大脑连上外面的世界。

**权力让渡的边界：** 系统不仅管权限，还主动帮模型积累经验。模型不需要自己记住一切——系统帮它在多个时间尺度上记忆。

**记忆系统：** 分层。第一层 MEMORY.md（类似 Claude Code）。第二层 SQLite + FTS5 全文搜索，所有历史会话可检索和摘要。第三层 8 个外部 memory provider 插件，其中 Honcho 把用户和 AI 都建模为 peer，做跨会话辩证推理。

**程序性记忆：** 这是 Hermes 最独特的设计。Agent 可以通过 `skill_manage` 自己创建、更新、删除技能。解决了一个非平凡任务后，它把方法保存为技能文件，下次复用。积累 20+ 自创技能后，同领域任务速度提升 40%。Hermes 甚至有一个 `hermes-agent-self-evolution` 仓库，用 DSPy 和 GEPA 做自动化技能优化——相当于给 Agent 加了一个慢速但持续的自然选择过程。

**核心答案：** 权力不要全交给模型——系统替模型积累经验，让能力在时间维度上复利增长。

### 2.3 OpenClaw：做管道，可插拔，不替模型做决定也不替模型记东西

OpenClaw 走的是第三条路：**信息管理的需求会变，与其内置一套固定方案，不如把接口留好，让生态去填。**

它的核心是 Gateway——一个 long-lived Gateway 拥有所有 messaging surfaces。CLI、macOS app、Web UI、自动化任务，全部通过 WebSocket 连接 Gateway。手机、Mac、远程设备通过同一个协议接入，声明自己是 node，暴露摄像头、屏幕、Canvas 等能力。你从 Telegram 发消息，从 Web UI 操作，从手机节点触发——最终进入同一套协议、同一套会话管理、同一套工具策略。

**权力让渡的边界：** 系统不替模型做决策，也不替模型记东西，但系统确保模型在任何入口、任何设备上都有相同的运行环境。**OpenClaw 先做了神经系统，而不是大脑。**

**Context Engine：** 可替换的运行时插件，四个生命周期点——ingest（消息进来）、assemble（构建上下文）、compact（压缩）、after turn（一轮结束）。内置的是 legacy engine，但插件可以注册完全不同的 engine。

**扩展性：** 把能力扩展拆成三层——内置 Tools（预装能力）、Skills（教 Agent 什么时候用、怎么用外部工具）、Plugins（打包前两者，一条命令安装）。ClawHub 上架了 58000 多个社区 Skill。

**面对模型进步：** OpenClaw 的基础设施层几乎不受模型进步影响。Gateway 不会因为 Claude 变强就不需要了，多入口路由不会过期，设备节点不会过期，会话隔离不会过期。模型再强，连接现实世界的管道也不会自动出现。

**核心答案：** 权力不交给模型也不集中在系统——权力分散在可插拔的接口和生态里。

### 2.4 三条路线，同一个问题

三条路线对"Harness 该管多少"的回答，可以用一张表看透：

| 维度 | Claude Code | Hermes Agent | OpenClaw |
| --- | --- | --- | --- |
| **权力分配** | 模型多拿，系统少管 | 系统帮记，模型复用 | 接口开放，生态填充 |
| **核心定位** | 权限系统 + 上下文管理 | Agent Loop 能力密度 | Gateway 多入口收束 |
| **记忆方案** | 极简：CLAUDE.md + Auto Memory | 分层：MEMORY.md → SQLite → 8 个外部 provider | 可插拔：Context Engine 四生命周期点 |
| **对模型进步的态度** | 做减法——每个组件应能被移除 | 做加法——经验积累不过期 | 做管道——基础设施不随模型变 |
| **一句话总结** | 模型自己行，系统别添乱 | 光靠模型不够，系统帮你记 | 别猜未来，把接口留好 |

有意思的是，三条路正在收敛。Claude Code 加了 auto memory 和 channels，Hermes 加了 Gateway 和多平台适配，OpenClaw 在做 Context Engine 可插拔化。目的地越来越像：一个有记忆、有技能、有入口、有权限、有持久状态的个人 AI 系统。

**而一旦 Harness 立住了，下一个问题自然浮现：既然单次运行的边界已经划清楚了，人能不能连"每次启动运行"这件事也交给系统？**

这就是第四次让渡——Loop Engineering。

## 三、第四次让渡：Loop —— 人把"什么时候做、按什么顺序做"也交了出去

### 3.1 Loop 在权力让渡弧线上的位置

回到 Cherny 的时间线：2025 年的标志性动作是"你写循环来提示 Claude"。2026 年则是"你构建驾驭系统来运行这些循环"。

Harness（驾驭系统）解决的是**单次运行的边界和约束**：Agent 在什么环境里跑？能用什么工具？不能动什么？跑完怎么验证？

Loop 解决的是**多次运行的编排和自动化**：什么时候该跑？跑什么任务？任务从哪来？结果怎么处理？下一轮跑什么？

Loop 站在 Harness 的肩膀上。没有 Harness 划定边界，Loop 就是让 Agent 在无约束的状态下无限自动运行——这是灾难。有了 Harness，Loop 才安全。

### 3.2 Loop 的六大构件

Addy Osmani 在他引发全球热议的《Loop Engineering》中，将 Loop 拆解为六个构件。每个构件都对应着一次**更细粒度的权力让渡**：

**① Automations（自动化）——"什么时候跑"交出去**

定时触发，让 Loop 从"你跑了一次"变成"系统按节奏持续跑"。Claude Code 中通过 `/loop` 定时、`/goal` 设定目标条件（如"test/auth 下所有测试通过且 lint 干净"）持续迭代直到满足。关键细节：验证是否完成的是一个**独立的小模型**，不是写代码的那个 Agent 自己——这是第六节会展开的核心原则。

**② Worktrees（工作树）——"在哪跑"的隔离**

多个 Agent 并行时，文件冲突是最大失败源。Git worktree 给每个 Agent 独立的工作目录，共享仓库历史但不共享文件。Codex 内建，Claude Code 提供 `--worktree` flag 和 `isolation: worktree` 设置。

**③ Skills（技能）——"怎么跑"的项目知识外部化**

Agent 每轮都是"冷启动"，会用自信的猜测填补信息缺口。Skill 就是把项目约定、编码规范、历史教训一次性写在外面，Agent 每次运行都读取，不再重新推导。没有 Skills，Loop 每轮循环都在零重新推导项目知识；有了 Skills，信息在循环之间**复利增长**。Hermes 更进一步——Agent 完成任务后把方法保存为新 Skill，积累 20+ 自创技能后同类任务速度提升 40%。

**④ Connectors（连接器）——"能看到什么"的拓展**

基于 MCP 协议，让 Agent 能触及文件系统之外的信息源——Issue 系统、数据库、CI/CD、Slack。没有 Connectors 的 Loop 只能看代码，有了 Connectors 才能实现"发现问题→修复→PR→通知"全闭环。

**⑤ Sub-agents（子代理）——"谁写谁验"的分离**

这是 Loop 质量的关键杠杆。**写代码的 Agent 和验证代码的 Agent 不应该是同一个。** 让编写代码的模型评判自己的代码，就像让学生给自己的考试打分一样不可靠。

Anthropic 内部数据：标准对话消耗 1x token，单 Agent 循环消耗 ~4x，多 Agent 系统消耗 ~15x——但多 Agent 在内部评估中比单 Agent 好了 90.2%。

**⑥ State（状态）——"上次跑到哪了"的跨会话记忆**

**这是 Loop 最核心的设计原则：模型会遗忘一切，但仓库不会。** 所有状态存在外部系统——STATE.md、Issue 系统、Git 仓库——不在上下文窗口里靠记忆。每个循环迭代从**全新上下文窗口**开始，但基于持久化的内容工作。

这就是最原始的 Ralph Loop 只用一行 `while :; do cat PROMPT.md | claude-code; done` 就实现永续自动化的原因：每次循环重新读取 PROMPT.md 和当前代码库状态，完全忽略之前对话历史。简单粗暴，但确实解决了长对话上下文的所有问题。

### 3.3 一个完整的 Loop 长什么样

把这六个构件组装起来：

1. 每天早上，一个 **automation** 在仓库上运行。
2. 它调用一个 triage **skill**，读取昨天的 CI 失败、open issues、最近 commits。
3. 发现的问题写入 **state** 文件。
4. 对每个值得处理的问题，开隔离 **worktree**。
5. 一个 **sub-agent** 在 worktree 里起草修复。
6. 另一个 **sub-agent** 根据 skills 和现有测试审查修复。
7. **Connectors** 自动开 PR、更新 ticket。
8. 处理不了的问题落入 triage 收件箱。
9. **State** 记住今天试了什么、过了什么、还剩什么。
10. 明天早上，Loop 从昨天停下的地方继续。

这就是 Cherny 所说"my job is to write loops"的具体展开——你不再逐个处理 issue，你设计了一个系统让它替你处理。

## 四、让渡到极致时，三条不可逾越的原则

当权力让渡走到 Loop 这一步——人在物理上已经不在场了——有三条原则决定了 Loop 是可信任的还是危险的。

### 4.1 自验证：机器信号，不是人的感觉

Agent 必须能端到端验证自己的工作——跑测试、跑 lint、跑类型检查。"我觉得改好了"不算验证，只有机器可检查的通过/失败信号才算。OpenAI 的验证闭环、Anthropic 的 Headless Chrome 验证，都是这条原则的工程实现。

### 4.2 Maker/Checker 分离：不让同一个大脑既当运动员又当裁判

写代码的 Agent 和验证代码的 Agent **必须是不同的实例**。Anthropic 在工程博客中分享了一个三 Agent 修复循环，用 GAN（对抗生成网络）的思路设计了三个角色：

- **Planner**：接收 1-4 句话的问题描述，扩展成修复方案。被**明确限定只关注"做什么"，不涉及具体实现**——因为 Planner 在技术细节上的错误会向下游级联。
- **Generator**：按方案逐步实现，完成后自评估，做出战略性决策：分数向好则继续优化，方向本身有问题则果断推翻。
- **Evaluator**：通过 Playwright MCP 像真实用户一样操作应用——点击 UI、测试 API、检查数据库。**任何一个维度低于阈值则整轮失败。**

Evaluator 能发现非常具体的 bug，比如：

> "矩形填充工具只在拖拽起止点放置了 tile，没有填充整个区域——`fillRectangle` 函数存在但在 `mouseUp` 时没有被正确触发。"
> "`PUT /frames/reorder` 路由定义在 `/{frame_id}` 之后，FastAPI 把 'reorder' 当作 frame_id 整数解析，返回 422。"

这种精确反馈，靠模型自我批评是得不到的。用 Anthropic 原文的说法——让模型评判自己的工作，它会"自信地称赞自己的工作，即使质量明显平庸"。

**Claude Code 的 `/goal` 命令在底层做的正是这件事：用一个独立小模型判断是否达成目标，不是让执行任务的模型自己判断。** Maker/Checker 分离是 Harness 和 Loop 设计中最不可妥协的结构性决策。

### 4.3 熔断机制：当 Agent 不再产生新信息时，切断比继续喂信息更重要

每个循环必须有硬性停止条件。生产参数：

- 最大迭代：15–25 步
- 单轮超时：~300 秒
- 单次成本预算：~$2

实用技巧：**对每轮迭代的动作取哈希指纹，连续 3 次相同立即停止。** 指纹重复说明 Agent 已陷入死循环，再喂信息也无济于事。从权力让渡视角看，这是一条"负向边界"：**当 Agent 不再产生新信息时，人对系统的信任就应该立即撤回。**

Lance Martin（Anthropic）说得一针见血：

> **"Rubric design is the skill now, the model is the easy part."**（设计可验证的成功/停止标准，才是今天真正的工程能力。）

## 五、Harness 衰变定律：让渡出去的权力，有些会自己回来

Anthropic 的核心研究员总结出了一个反直觉的趋势：

> **模型原生能力 ∝ 1 / 所需 Harness 的复杂度**

在 Claude 3.0 时代，为了不让 Agent 跑偏，Harness 需要极度厚重的约束：硬编码流控、频繁截断并重置上下文、密密麻麻的正则检查。但模型升级到 Claude 3.5 之后，开发者们惊讶地发现：过去需要靠外围 Harness 代码硬卡死的逻辑，模型自己就轻松搞定了。

**这意味着：有些你让渡给系统的权力，随着模型进步，系统应该把它们还给模型。**

一个高段位的 AI 工程师，会把 Harness 收拢在两类**不会被模型进步消解**的场景中：

1. **绝对的合规与业务硬边界**：金融风控红线、权限隔离（Sandbox）、财务审批流（Human-in-the-loop）。这些涉及现实世界法律与责任的边界，哪怕 AI 再聪明，也必须由 Harness 硬性卡死——因为这里不是能力问题，是责任问题。
2. **外部环境的互操作系统**：无论模型演进到何种形态，它永远只是个思考内核。它无法自行拥有生产环境的 API 密钥和数据库连接。提供安全受控的工具调用和状态存储，是 Harness 的永久护城河。

**权力让渡不是单向的。模型每变强一次，人都应该回头审视：有哪些我当初不信任模型所以交给系统管的东西，现在可以交还给模型了？**

## 六、从零到 Loop：渐进式让渡的实践路径

权力让渡不需要一步到位。渐进式来。

### 6.1 第一站：`/goal` —— 把"怎么做到"交出去

在 Claude Code 里，最简单的循环化：

```
/goal: 所有 Playwright 截图与设计稿的像素差异 < 1%，且 npm test 全部通过
```

设好目标，走开。Agent 自己修改代码、跑测试、看结果、再调整。`/goal` 的价值在于：它让你从"逐步指挥 Agent 该做什么"转变为"定义终态，让 Agent 自己找路"。而验证终态的是一个独立小模型——Maker/Checker 分离内置其中。

### 6.2 第二站：并行 + 隔离 —— 把"互不干扰"的保证交给系统

用 `--worktree` 让多个任务在隔离的 checkout 中同时运行。一个 Agent 修 auth 模块的 bug，另一个给 settings 加功能，互不干扰。

### 6.3 第三站：Maker/Checker 分离 —— 把"验证"交给另一个大脑

在 `.claude/agents/` 里定义 reviewer 子 Agent。安全审查器关注漏洞，性能审查器关注瓶颈，规范审查器关注编码风格。

### 6.4 第四站：持久化状态 —— 把"记忆"交给外部系统

把每次运行的结果、踩过的坑写入 `PROGRESS.md`，让下一轮从上次的进度开始。这就是 outer loop 的雏形。

### 6.5 成本控制：权力让渡不是免费的

- 单 Agent 循环 token 消耗约为标准对话的 **4 倍**，多 Agent 约为 **15 倍**。
- Prompt caching 可让成本从 $3/百万降到 $0.30/百万（**10 倍差距**）。保持 prompt 追加式增长，不要中途改 tools 列表或模型——会破坏 cache。
- Anthropic 三 Agent harness 单次运行 $125-200（生成完整应用）。日常 bug 修复循环应在 **$2 以内**。
- 设好 token 预算上限。

### 6.6 常见陷阱

- **少而精的工具集 > 大而全的工具库**：如果一个人类工程师都说不清该用哪个工具，Agent 更做不对。
- **暴露错误，不要吞掉**：工具返回错误字符串而不是抛异常——异常中断循环，错误字符串让模型有机会自纠错。
- **保留失败痕迹**：擦除失败等于擦除证据，模型会重复同样的错误。

## 七、让渡的尽头：人还剩什么？

Addy Osmani 在文章结尾给出了三条警告。从权力让渡的视角重新读它们：

### 7.1 验证仍然是你的责任——"信任但核实"的权力不能交

Agent 说"done"只是一个声明，不是一个证明。循环跑完、CI 全绿、PR 自动开了——这些只代表你设定的检查通过了，而你的检查可能不够好。

**最终验证权，是人永远不应该让渡出去的权力。**

### 7.2 理解力债务——权力让渡的隐藏代价

Loop 越快交付你没写过的代码，你的代码库中"存在但你不理解"的部分就越大。Addy 称之为 **comprehension debt（理解力债务）**——一个平滑的 Loop 只会让它增长得更快，除非你主动去读 Loop 产出的代码。

**每让渡一次执行权，你就要多花一分理解成本来追回认知。**

### 7.3 认知投降——权力让渡最隐蔽的陷阱

当 Loop 自动运行时，你很容易停止拥有自己的判断，直接接受它给出的任何东西。Addy 称之为 **cognitive surrender（认知投降）**。

两个不同的人可以构建完全相同的 Loop，得到截然相反的结果。一个人用它来在自己深度理解的工作上走得更快。另一个用它来回避理解工作本身。Loop 不知道区别。你知道。

> "Build the loop. But build it like someone who intends to stay the engineer, not just the person who presses go."

**这就是 Loop 设计比 Prompt 工程更难的真正原因——不是技术更难，而是自律更难。权力让渡出去之后，你还需要有足够的纪律把它拿回来审视。**

## 八、结语：人类掌舵，Agent 执行

从 Prompt 到 Context 到 Harness 到 Loop，每一次进化都是一次权力让渡：把"怎么写"交给 AI、把"看什么"交给检索系统、把"在什么边界内运行"交给 Harness、把"什么时候跑什么任务"交给 Loop。

但有一条权力从来没有、也不应该被让渡：**决定让渡什么、让渡到哪里的判断力。**

OpenAI 在其数百万行代码的实验结语中留下了全行业最值得铭记的一句话：

> **"Human steer, agents execute."**（人类掌舵，Agent 执行。）

软件工程没有消亡。但工程师的核心价值已经完成了三次升维：

1. **定方向（Steering）**：精准定义问题和终局形态——这种高维度产品与架构思维，是模型最难替代的。
2. **搭架子（Harnessing & Looping）**：为 Agent 编排运行支架——干净的上下文源、严密的自动化流水线、让报错自动闭环。
3. **做判别（Decision Making）**：在关键的技术交汇点和高风险边界，引入人类介入，一锤定音。

如果你今天还在每次打开 Claude Code 都从头写提示词，试试 `/goal`。如果你已经在用 `/goal`，试试 Dynamic Workflows。如果你已经有了多 Agent 系统，试试把它变成每天自动运行的 automation。

不需要一步到位。循环的美妙之处在于——它就是为了迭代而设计的。而工程师的美妙之处在于——**每一次迭代之后，你都有权力重新审视：哪些权力该收回来，哪些该放出去。**
