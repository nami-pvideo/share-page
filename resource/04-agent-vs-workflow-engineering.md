# Agent vs Workflow：两种 AI 模式的选择与工程化

> "让 AI 帮我改 bug"和"每天凌晨拉取数据生成日报"——这两件事看起来都是在"用 AI"，但它们适合的模式完全不同。前者需要 Agent 的自主探索能力，后者需要 Workflow 的确定性执行。本文深入剖析两种模式的底层差异、各自适合的场景、以及如何在工程环境下把这两种模式组合起来。

---

## 一、原理篇：两种模式的底层架构

### 1.1 Agent 模式：自主决策循环

Agent 模式的核心是一个**思考-行动-观察的闭环**：

```text
[接收目标] → [思考] → [调用工具] → [观察结果] → [再思考] → ...
    ↑                                                        │
    └────────────── [决策：目标达成/需要更多信息] ←────────────┘
```

每一次迭代，AI 都会：
1. 根据当前拥有的信息判断：还需要什么？下一步做什么？
2. 调用工具获取新信息（读文件、跑命令、搜索代码库）
3. 评估结果：信息够了没？方向对不对？
4. 决定继续探索还是输出最终结果

这个循环的**核心特征是自主性**——AI 决定每一步做什么。你只需要给一个目标，它自己找路。

市面上的主流产品几乎都是 Agent 模式：
- **Claude Code**：命令行工具，完全自主地读文件、写代码、跑测试
- **Cursor**：编辑器内的 Agent，可以跨文件操作
- **Aider**：命令行 Agent，自主编辑代码、管理 git
- **Devin**：全栈 Agent，可以独立完成从分析到部署的完整开发流程

### 1.2 Workflow 模式：确定性 DAG

Workflow 模式的核心是一个**预定义的步骤序列**：

```text
[触发条件] → [步骤1: 数据采集] → [步骤2: 模型推理] → [步骤3: 结果存储]
               输入: api_key        输入: prompt          输入: db_conn
               输出: raw_data       输出: result          输出: success/fail
```

每一步：
- 输入和输出类型在编排阶段就确定了
- 缺少任何一个输入参数，步骤直接报错
- 没有"自主判断"环节——它不会说"这个参数看起来不对，我换一个试试"
- 分支条件必须在编排时就写清楚

代表产品：
- **n8n**：可视化编排自动化工作流，支持数百种集成
- **Dify**：LLM 应用编排平台，以可视化方式搭建 AI 工作流
- **Make（原 Integromat）**：自动化编排工具
- **Airflow / Temporal**：面向数据工程的工作流引擎

### 1.3 核心差异对比

| 维度 | Agent 模式 | Workflow 模式 |
|------|-----------|--------------|
| **控制权** | AI 主导每一步决策 | 人类预定义步骤顺序 |
| **信息要求** | 可以缺，缺了自己找 | 一步不能缺，缺就报错 |
| **可预测性** | 低——每次探索路径可能不同 | 高——路径固定 |
| **灵活性** | 高——能应对意外情况 | 低——只能处理预定义的分支 |
| **出错方式** | 静默走偏，不易发现 | 明确报错，容易定位 |
| **自动化程度** | 低——通常需要人类 Review | 高——配置好就无人值守 |
| **每次成本** | 高——探索过程消耗大量 Token | 低——每次执行的 Token 消耗固定且可控 |

## 二、选型篇：什么时候用什么

### 2.1 选型决策矩阵

| 任务特征 | 推荐模式 | 理由 |
|---------|---------|------|
| 边界不清晰，需要跨多个文件/模块定位 | **Agent** | 探索是必要的成本 |
| 一次性任务，迭代式逼近目标 | **Agent** | 编排成本不值得 |
| 涉及创造性产出（写文档、设计方案） | **Agent** | Workflow 表达不了创造逻辑 |
| 流程固定、输入输出明确、需定期执行 | **Workflow** | 编排一次，永久受益 |
| 需要严格的审批和审计 | **Workflow** | 每一步可追踪、可审计 |
| 对稳定性有极高要求 | **Workflow** | 不做"意外的事" |
| 需要集成多个外部系统 | **Workflow** | 字段映射用 DAG 表达最清晰 |

### 2.2 选错模式的代价

**把 Workflow 任务交给 Agent**：

一个数据日报任务如果每次都用 Agent 来跑——它每次都会重新"探索"数据源、重新"理解"报表结构、重新"决定"展示格式。结果：
- 每次 Token 消耗不固定，而且通常偏高
- 每天的输出格式可能不完全一致
- 如果某天 Agent "理解"错了某个字段，输出会有偏差

**把 Agent 任务塞进 Workflow**：

一个 Bug 修复任务如果试图用 Workflow 表达——需要预定义每一步要读哪个文件、改哪一行、怎么验证。但 Bug 定位本身就是探索过程，Workflow 做不到"先看看这个错误出现在文件 A 还是文件 B，然后决定下一步查哪里"这种动态分支。

**选型口诀**：重复的事用 Workflow，探索的事用 Agent。

### 2.3 混合模式：业界实践中两者如何协作

在实际项目中，Agent 和 Workflow 往往不是"二选一"，而是各司其职：

**模式一：Agent 开发 + Workflow 交付**

日常开发中，用 Agent 完成探索和编写——改 Bug、加功能、写测试。当功能稳定后，把其中可重复的部分沉淀为 Workflow，实现自动化执行。

例如：团队的 CI/CD 管线里，代码审查阶段先由 Agent 跑一轮自动 Review（探索型），再进入人工 Review 阶段。部署后的监控报警由 Workflow 处理（确定型）。

**模式二：Workflow 框架 + Agent 模块**

在 Workflow 中嵌入 Agent 作为其中一个步骤。n8n 和 Dify 都支持这种模式：

```text
[定时触发] → [拉取数据库] → [Agent 分析异常数据] → [生成报告] → [发送通知]
             确定型步骤        探索型步骤              确定型步骤     确定型步骤
```

Agent 模块只在需要"理解和判断"的环节介入，前后步骤由确定性逻辑保证。

**模式三：Agent 编排 Workflow**

Claude Code 和 Cursor 等 Agent 工具可以调用外部 Workflow——让 Agent 决定"现在需要启动这个定时任务"。这本质上是把 Agent 当作一个"智能触发器"。

## 三、基础设施篇：Agent 模式的工程化支撑

Agent 模式虽然灵活，但它的"走偏"风险需要通过工程化手段来管理。以下四个基础设施是支撑 Agent 模式在工程环境下稳定运行的关键。

### 3.1 Harness：把 AI 安全接入本地的"夹层"

Harness 不是 AI 本身，而是 AI 模型和本地代码之间的适配层。它做的事包括：

| 能力 | 作用 | 举例 |
|------|------|------|
| 上下文注入 | 每次会话自动加载项目文档 | 自动读取 CLAUDE.md、当前打开的文件 |
| 工具调度 | 让 AI 能调用本地工具 | Read、Bash、Edit、Git 等 |
| 权限控制 | 敏感操作需人工确认 | 首次改文件需要 approve |
| Hook 系统 | 工具调用前后插自定义脚本 | 禁止写敏感目录、跑命令前备份 |

**Claude Code 的 Hook** 是一个典型例子：你可以配置 `PreToolUse` hook，在 AI 试图修改某个目录时自动拦截并提醒。Cursor 的 Rules 系统也提供类似的项目级行为约束。

把 Harness 配置好，就是在给 AI 的 Agent 模式上加上"安全围栏"。

### 3.2 Git Worktree：为每个 Agent 提供隔离空间

当多个 Agent 并行工作时，最头疼的问题就是文件状态冲突。A 任务改到一半的文件，被 B 任务覆盖——这在单工作目录下几乎是不可避免的。

**Git Worktree** 是一个简单但强大的解法：

```bash
# 创建独立工作目录
git worktree add ../project-feat-a feat/a
git worktree add ../project-feat-b feat/b
git worktree add ../project-bugfix  bugfix/issue-42

# 每个 worktree 有独立的分支和文件状态
# Agent 在各自 worktree 里工作，互不干扰
```

效果相当于给每个 AI 任务分配了一张"干净的桌子"：

```text
~/code/project/              ← 主仓库（做 Review）
~/code/project-feat-a/       ← Worktree A（Agent 1 做新功能）
~/code/project-feat-b/       ← Worktree B（Agent 2 做新功能）
~/code/project-bugfix/       ← Worktree C（Agent 3 修 Bug）
```

这是实现"一人多需求并行"的物理基础。不需要 stash 来 stash 去，不需要担心 Agent 互相覆盖文件。

### 3.3 Skill 系统：给 Agent 装"资深同事的 SOP"

即使给了 Agent 充足的信息和安全的执行环境，它仍然有一个倾向——"跳过关键步骤"。

不加约束时，一个 Agent 接到"写个新功能"的任务可能会：

1. 直接写代码，跳过设计讨论
2. 不写测试
3. 写完之后声称"完成了"，但没有跑过任何验证

**Skill 系统**就是用来对抗这种倾向的。它是一组成文的 SOP（标准操作流程），告诉 AI 在某些关键节点必须怎么做。

业界实践：

- **[obra/superpowers](https://github.com/obra/superpowers)**：一套 Claude Code 的 Skill 集合，包含 brainstorming → writing-plans → executing-plans → verification 的四步主线，以及 /tdd、/diagnose、subagent-driven-development 等场景化 Skill
- **[mattpocock/skills](https://github.com/mattpocock/skills)**：强调领域语言沉淀，通过 CONTEXT.md 和术语表让 AI 理解项目特定的概念体系
- **Claude Code 内置 Hooks 和 CLAUDE.md**：允许团队自定义 Skill，覆盖从编码规范到验证规则的各类约束

核心的四步骨架：

```text
brainstorming → writing-plans → executing-plans → verification
  先想清楚        拆成小步骤        按步骤实施          展示证据
```

这套流程不是让 AI 更聪明——是让它在关键节点上不跳步骤。

### 3.4 文档中心：AI 的长期记忆

Agent 每次会话都从零开始——这是它最大的局限之一。解决的方案是把"团队共识"落进持久文档：

```text
项目根目录/
├── CLAUDE.md              ← AI 行为约束（自动读取）
├── CONTEXT.md             ← 领域术语表
├── README.md              ← 项目概述
└── docs/
    ├── prd/               ← 需求文档（持续累积）
    ├── adr/               ← 架构决策记录
    └── api/               ← 接口文档
```

关键不在于文档的存在，而在于**持续累积**。每次做完一个需求，把新增的概念、边界、坑点追回对应的文档里。下次 Agent 再碰这个模块，一份文档就拿到完整上下文。

## 四、总结

Agent 和 Workflow 是两种互补的模式。核心总结：

| | Agent | Workflow |
|---|------|------|
| **一句话** | "你自己想想怎么干" | "按照步骤一步一步来" |
| **适合** | 探索型、一次性、创造性任务 | 重复型、确定性、无人值守任务 |
| **风险** | 自由扩散、走偏难发现 | 灵活性不足、无法应对意外 |
| **工程支撑** | Harness + Worktree + Skill + 文档中心 | 编排平台 + 监控 + 告警 |

**选型口诀**：重复的事用 Workflow，探索的事用 Agent。日常开发 95%+ 走 Agent 模式，定时/批量/数据搬运走 Workflow 模式。

两者的组合才能让 AI 在工程环境下稳定、高效的运转。
