# 上下文管理：AI 工作内存的优化策略

> 设想一个场景：你把整个仓库的日志、配置、文档一股脑塞给 AI，让它分析 bug。AI 认真"分析"了一圈，给了你一份漂亮的分析报告。但你看完发现，它从头到尾都没打开过那个真正包含 bug 逻辑的源文件。问题不在 AI 不努力——问题在它那块工作内存被噪音占满了。本文系统性拆解上下文窗口的工作机制，以及如何高效管理这块 AI 协作中最稀缺的资源。

---

## 一、原理篇：context window 是什么

### 1.1 工作内存模型

把 AI 的上下文窗口理解为它的**工作内存**。这块内存有两个硬约束：

1. **容量上限**：以 Token 计数，几十万到几百万 Token 不等（不同模型不同），但不是无限。
2. **会话独占**：每次会话是独立的工作空间。前一次会话里 AI"学"到的项目知识，不会自动带到下一次会话。

在一次会话中，以下所有内容都在消耗这块内存：

| 内容来源 | 消耗位置 | 是否可控制 |
|---------|---------|----------|
| 系统提示词和工具注入 | 会话开头（自动） | 部分可配置 |
| 工程师说的每一句话 | 对话记录 | 可控 |
| AI 生成的每一句回答 | 对话记录 | 间接可控 |
| 工具调用结果（读文件、跑命令等） | 对话记录 | 可控 |
| 会话历史 | 对话记录 | 累积中 |

当说话说到一半，你让 AI "再优化一下刚才写的那个函数"——它能"想起"那个函数，是因为整段对话还在上下文窗口里。

当上下文窗口被填满，发生的事情取决于工具实现：
- 部分工具会自动**压缩历史**（summarize 旧对话），但压缩必然丢失细节。
- 部分工具会直接**截断**，AI 开始"失忆"——它忘了最开始定义的验收标准、忘了之前发现的边界情况。

### 1.2 隐性代价：不只是容量上限

上下文窗口的容量是一回事，它的隐性代价是另一回事。很多人只看到了"能塞多少"，忽略了"塞满之后发生了什么"。

**代价一：注意力稀释**

上下文越长，AI 越容易"看花眼"。就像一个工程师面前摆了 50 个文件同时阅读，他可能每个都看了，但关键的那一段反而被埋在了第 37 页。AI 也是——给它看 5000 行日志和 3 个核心代码文件，它可能"认真分析"了日志的每一个 WARNING 级别条目，然后基于日志里的关键词推测代码路径，但始终没打开那段真正出错的代码。

**代价二：Token 成本**

对于基于 API 调用的工具，上下文越长，每次请求的 Token 消耗越高。这会直接推高使用成本。对于本地模型，这意味着更多的 GPU 显存消耗。

**代价三：缓存命中率下降**

很多 AI 平台对前缀有 prompt cache 机制——如果两次请求的上下文前缀相同，后续请求可以复用缓存，大幅降低成本并提高速度。但上下文越长，越容易被插入的新内容"打破"缓存的前缀匹配，导致缓存持续失效。

**核心认知**：上下文窗口不是"能塞多少塞多少"的免费资源，而是要精打细算的稀缺品。

### 1.3 上下文窗口在不同工具中的差异

不同 AI 编码工具的上下文管理策略差异很大：

| 工具 | 上下文窗口 | 管理策略 |
|------|----------|---------|
| Claude Code | 200K tokens（Claude 3.5 及后续） | 自动压缩旧对话、文件智能分块读取 |
| Cursor | 取决于模型 | 智能上下文筛选，基于光标位置和最近编辑自动注入 |
| GitHub Copilot Chat | 取决于模型 | 基于当前打开文件+选中文本构建上下文 |
| Aider | 取决于模型 | 使用 map-reduce 分块处理大文件的 diff |
| Cline | 取决于模型 | 手动上下文管理，支持 MCP 按需拉取 |

关键差异：有些工具**主动帮你选上下文**（Cursor 只把当前文件和最近编辑注入），有些工具**把管理权限交给你**（Aider 需要你指定文件范围）。理解你所用工具的策略，才能用好它的上下文窗口。

## 二、实践篇：三种信息流入策略

### 2.1 主动塞（Push）

由工程师直接把文件内容、需求描述、关键代码片段贴进对话。

**优点**：
- 精准控制——AI 第一时间就能基于你选的信息开始工作
- 减少 AI 自主探索中的错误判断——你不用等它自己 grep 完整个仓库才发现它的搜索范围不对

**缺点**：
- 得知道"塞什么、塞多少"——塞错了既占内存又误导 AI
- 一次性塞进去的内容无法动态调整

**适用场景**：
- 对话开始时塞入核心上下文：PRD + 2-3 个核心代码文件 + 明确的边界声明
- 关键定义：术语表中与本次任务相关的条目
- 示例参照：类似的已有实现，作为风格的参照系

### 2.2 被动探索（Let Agent Search）

只说目标，让 Agent 自己去 grep、读文件、跑命令，自主搜集它认为需要的信息。

**优点**：
- 省工程师的力气——你不用替它列出所有相关文件
- 开放性问题的自然解法——你自己也说不清问题出在哪个文件时，让 AI 去做第一轮定位

**缺点**：
- "探索"本身消耗 Token——每次 grep 结果、read 内容都会写进上下文
- 可能误判——读到过时的文件、把无关模块当主线
- 可复现性差——每次探索路径可能不同

**适用场景**：
- 问题定位不明确时的第一轮诊断
- 需要跨多个未知文件收集信息的开放性问题
- 一次性任务，不值得专门写指令列出文件清单

### 2.3 工具按需取（Pull on Demand）

通过 MCP（Model Context Protocol）等接口，让 AI 在需要时自己去拉取信息。

**优点**：
- 按需付费——不预先占上下文窗口
- 内容可以很大——API 文档可能有几千行，AI 只需要其中一段；拉进来了也只占一小块上下文

**缺点**：
- 需要额外的工具配置
- AI 需要知道"什么时候该去拉"——这本身需要提示词的引导
- 多一次工具调用，多一步延迟

**适用场景**：
- 可能用到但不一定用的辅助信息：API 文档、数据库 Schema、配置中心的规则
- 规范化的外部知识库：内部平台的接入文档、CI/CD 流程说明
- 需要实时数据：当前系统的状态、最近的部署记录

### 2.4 实战配方：混合策略

三种方式在实际使用中不是互斥的，而是分层组合：

```text
[会话开始]
    ↓
[策略一：主动塞]
    塞入 PRD 核心段落 + 2-3 个主链路文件 + 本次明确的边界
    目标：把方向定准，AI 不需要在"我该看什么"上浪费 Token
    ↓
[策略二：被动探索]
    AI 在框定的起点上自主推进，读取它认为相关的代码
    目标：在可控半径内让 AI 发挥自主性
    ↓
[策略三：工具按需取]
    遇到特定问题时按需拉取文档、配置、知识库
    目标：细节按需获取，不预占内存
```

这套配方的设计思想：**前期把方向定准，中期让 AI 在安全半径里自己跑，后期按需补细节**。

## 三、反模式篇：上下文污染的八个高发场景

### 3.1 一次性塞入大量日志

**典型操作**：把应用运行日志一整个文件复制粘贴进对话，说"帮我看一下有没有异常"。

**后果**：几千行日志里可能只有 2-3 行关键错误，被淹没在大量 INFO 和 DEBUG 级别的条目中。AI 可能在逐行分析日志上耗尽上下文，却始终没打开实际的源代码。

**修正**：先用 shell 过滤出 ERROR/FATAL 行，只把有价值的上下文送进去。或者用 AI 的工具能力让它自己 grep——减少你手动分析的体力，但不让日志直接占满窗口。

### 3.2 把整个配置文件和 README 贴进对话

**后果**：这些内容 AI 完全可以自己 read。贴进去不仅占内存，还会分散它对真正关键的代码的注意力。

**修正**：让 AI 自己读文件。它的 Read 工具只会在调用后把内容写进对话——如果你不需要它立刻处理这段内容，就不要主动贴。

### 3.3 在长对话中不断追加新任务

**典型操作**：一个会话从上午开到下午，改了 5 个不同的功能，中间穿插各种讨论。

**后果**：旧任务的讨论、代码、错误处理全部占着上下文，新任务的关键信息反而被稀释。AI 可能把新任务的验收标准和旧任务的混淆。

**修正**：不同任务开不同会话（或不同 worktree 下的独立会话）。会话是有状态的，但不要让它变成"什么都在里面"的杂货铺。

### 3.4 主动塞入"可能用到"的所有文件

**典型操作**："可能 AI 会需要这些文件，先全给它看"——于是对话开头贴了 10 个文件。

**后果**：注意力稀释。AI 在 10 个文件中迷失关键信息。一个 3 文件就能解决的问题，多了反而做不好。

**修正**：只贴最核心的 2-3 个文件。AI 需要更多时，它自己会读。

### 3.5 在上下文接近上限时追加关键信息

**后果**：如果工具自动启动了历史压缩，你追加的内容可能被"挤掉"或"压缩掉"关键细节。

**修正**：在对话中期，如果感觉上下文快满了，主动清理解或开新会话。把当前最重要的结论总结出来，复制到新会话作为起点。

### 3.6 反复让 AI 重新读已读过的文件

**典型操作**：在同一个会话中，AI 之前已经读过 `service.go`，后来你又说"你再看看 `service.go` 的 XX 方法"。

**后果**：文件内容重复写入上下文，白白消耗空间。

**修正**：如果 AI 在当前会话中已经读过，可以直接引用函数名和行号，不需要让它再读整个文件。

### 3.7 使用超长 Prompt 模板

**后果**：有些团队会准备一份几百行的"AI 提示词模板"，里面包含大量背景信息、规则、最佳实践。每次用 AI 都把这份模板贴进去。几百行模板 + 实际任务内容，上下文窗口已经去了一小半。

**修正**：把长期有效的规则放到 CLAUDE.md 或 AGENTS.md 里（工具自动读取，不算在对话 Token 消耗中）。对话中只写与本次任务直接相关的背景。

### 3.8 忽略工具的上下文管理特性

**典型操作**：不确定工具怎么管理上下文，只知道"它能用"。

**后果**：不同工具的上下文管理策略差异巨大。如果工具会自动压缩旧对话，你可能不知道某些早期设定已经被压缩了；如果工具会自动注入当前文件，你可能不知道它"眼里的项目"其实只限于编辑器当前打开的那个文件。

**修正**：花 10 分钟了解你所用工具的上下文管理策略。Claude Code 有 `--verbose` 模式可以看 Token 消耗；Cursor 的上下文面板显示当前注入的文件列表。

## 四、总结

上下文管理就是 AI 协作中的内存管理。核心原则只有一条：

**塞进上下文的每一段内容，要么是 AI 立刻就要用的，要么是它接下来一定会问的。其他都是噪音。**

三个可立即执行的优化动作：

1. **主动塞不主动堆**：塞最核心的 2-3 个文件和明确的边界，不要塞"可能用到"的。
2. **一条会话一条事**：不要在一个会话里混着做不同方向的事。不同任务开不同会话。
3. **长期信息进配置，一次性信息进指令**：架构约束写 CLAUDE.md，本次需求的边界写在当前对话里。
