# AI 协作的心智模型：把 AI 当成新员工来带

> 为什么有的工程师用 AI 写代码越用越顺手，有的越用越头疼？差别往往不在技术层面，而在一个基础认知：你到底是把 AI 当"全知全能的代码生成器"，还是当"能力很强但什么都不懂的第一天入职员工"？本文从心智模型出发，提供一套可操作的"带新人"式 AI 协作方法论。

---

## 一、原理篇：为什么是"新员工"类比

### 1.1 AI 的真实能力剖面

在正式展开"新员工"类比的细节之前，先客观地看 AI 编码工具的能力剖面：

**AI 天然擅长的**：
- 编程语言的语法和标准库（Go 的 goroutine 怎么用、React 的 Hook 规则）
- 主流框架的常见用法（Gin 的中间件、MongoDB 的聚合管道）
- 设计模式（工厂模式、策略模式、Repository 模式）
- 代码转换（重构、格式化、语言间迁移）

**AI 天然缺少的**：
- 你的项目里"任务"和"工单"是两个东西还是同一个东西
- 为什么 Service 层绝对不能直接调 Handler 层
- 你们团队 Git 分支命名到底用 `feat/`、`feature/` 还是 `ft/`
- 这次改动的那些文件是属于 Provider 层还是 Gateway 层

**AI 有倾向但可被约束的特性**：
- 它倾向于"把所有相关的都改了"——而不是只改你要求的那一个
- 它倾向于给出"完整"的解决方案——即使问题只需要一个局部修改
- 它倾向于认为自己"做完了"——在没有跑过测试的情况下

### 1.2 候选类比的对比

为什么用"新员工"？和另外两个常见类比的对比：

| 类比 | 心理暗示 | 问题 |
|------|---------|------|
| **"外包"** | "我要什么你说清楚，别的我不管" | 暗示"甩需求"式协作。与 AI 协作需要的"指路+纠偏"模式相左 |
| **"实习生"** | "它会犯错，能力有限" | 低估了 AI 的真实能力——它写复杂代码的质量远高于实习生 |
| **"新员工"** | "能力扎实但不懂项目，需要带" | 中性且准确——不夸大也不贬低，定位为"需要 onboarding 的资深协作者" |

"新员工"类比的核心优势：

1. **能力匹配**：AI 和新员工一样"编程基本功扎实但不懂当前项目"
2. **行为可类比**：新员工需要什么（资料包、工位、师傅带、犯错后的纠正），AI 也需要类似的东西
3. **心理预期校准**：你不会期待新员工第一天就独立完成复杂任务，同样也不该期待 AI 一次就能做到

### 1.3 类比映射表：新员工的日常 → AI 协作的工程实践

| 新员工入职需要什么 | 映射到 AI 协作 | 工程实现 |
|-----------------|--------------|---------|
| 入职资料包（项目简介、术语、规范） | 信息分层补齐第二层（团队共识） | CLAUDE.md + CONTEXT.md + ADR |
| 一张干净的工位和电脑 | 独立的执行环境 | Git Worktree + 独立会话 |
| 师傅告诉TA"从哪开始看、哪些不能动" | 三段式指令（起点+边界+验证） | 每次派活时的明确指令 |
| 师傅的工位旁边可以随时问 | 对话式交互 | Agent 模式的思考-行动循环 |
| 公司的工作 SOP（怎么做需求、怎么发起 Review） | 标准化的 AI 协作流程 | Skill 系统（brainstorming → plans → execute → verify） |
| 犯错后师傅指正 | Review + 反馈 + 更新 CLAUDE.md | Code Review → 纠正错误模式 → 写入约束文档 |

这个映射表是整篇文章的骨架。下面逐个展开。

## 二、实践篇：像带新员工一样带 AI

### 2.1 给 TA 一份"入职资料包"

新员工入职第一件事是看资料。AI 的入职资料包就是项目根目录下的那几份文件：

```text
仓库规范/
├── CLAUDE.md / AGENTS.md      AI 行为约束文件
├── CONTEXT.md                 领域术语表
├── README.md                  项目简介
├── docs/
│   ├── prd/                   需求文档
│   ├── adr/                   架构决策记录
│   └── api/                   接口文档
```

**CLAUDE.md 怎么写**——三条原则：

1. 写**具体约束**，不写抽象原则。"关注点分离"是抽象原则；"Handler 层只做 HTTP 绑定，不含业务逻辑"是具体约束。
2. 优先写**禁止事项**。AI 默认会"扩散"，所以先告诉它"哪里不能碰"比告诉它"可以做什么"更关键。
3. 写**最短示例**。抽象规范配一个具体的、可执行的代码片段作为参照。

**CONTEXT.md 怎么写**——每条术语包含：

```markdown
## 订单（Order）
- 定义：用户提交的购买请求，包含 SKU、数量、金额、状态
- 状态流转：待支付 → 已支付 → 处理中 → 已完成 / 已取消
- 与其他概念的关系：
  - 订单 vs 工单：订单是用户侧的购买记录，工单是系统侧的处理记录
  - 订单 vs 退款单：退款单关联源订单，是对已支付订单的逆向操作
```

核心是**消除歧义**。AI 如果看到"订单"和"工单"两个词，它不知道这两者在你的系统里是不是同一个东西——除非 CONTEXT.md 说清楚了。

### 2.2 给 TA 分配任务时"指路"而不是"甩需求"

新员工第一天，你不会说"去优化一下订单模块"然后就走了。你会说：

> "你先看 `orders/service.go` 的 `CreateOrder` 函数，它在处理库存时少了一个并发锁。参考 `inventory/service.go` 里的锁实现，改完之后跑 `go test ./orders/... -run TestCreateOrder` 验证。"

这对 AI 也是完全适用的。一个好的 AI 任务指令，跟带新员工时的交代几乎一样：

```text
【起点】你看这次问题大概率在哪个文件/函数
【参照】类似的正确实现可以参考哪个文件
【边界】哪些文件/层/模块绝对不能改
【验收】跑什么命令如果通过了就算完成
```

这就是"指路"。你不是在替 AI 做事，你是在告诉它从哪扇门进去，以及哪些房间不要进。

### 2.3 给 TA 一张干净的桌子（并行任务不打架）

新员工需要一张独立的桌子——你不会让两个新员工共用同一张办公桌，他们的文件会混在一起。

AI 也一样。`git worktree` 就是给每个 AI 任务独立办公桌的命令：

```bash
# 为三个并发任务创建三个独立工作区
git worktree add ../repo-feat-a feat/new-feature
git worktree add ../repo-feat-b feat/refactor
git worktree add ../repo-bugfix bugfix/crash

# 三个 AI 会话，各自绑定到一个目录
# 互不干扰，各自在自己的分支上工作
```

这解决了多个 AI 任务并行时最头疼的问题——文件冲突和分支混乱。

### 2.4 发现 TA 走偏时"纠偏"而不是"放弃"

新员工犯错后，你不会说"算了，我自己来"。你会指出问题，让他重新做。

AI 也一样。如果它改了一些不该改的东西：

1. **指出问题**：告诉它具体错在哪——"你改了 Gateway 层的文件，但这次只需要改 Provider 层"
2. **更新约束**：如果这是系统性风险（它很容易再次犯同样的错），就把这条写进 CLAUDE.md——"Provider 相关改动不要触及 Gateway 层和 Hook 层"
3. **让它重做**：AI 的犯错成本远低于人类——让它回滚重新做你不会心疼

每次纠偏都是一次"Onboarding 质量提升"——你在锻炼 AI（更准确地说，你在优化它下一次会话时的信息基础）。

### 2.5 给 TA 一份"SOP"——不让它跳步骤

新员工可能会跳过 Code Review 就直接合代码、跳过写测试就直接写实现——不是故意的，是没人告诉 TA 流程。

AI 更会这样。如果你不约束它，它的工作路径很可能是：

```
接到任务 → 直接写代码 → 说"完成了"
   ↑                        ↑
  跳过了设计和对话           跳过了测试和验证
```

**Skill 系统**就是对抗这种跳跃的。核心四步：

```text
brainstorming → writing-plans → executing-plans → verification
  先聊清楚        拆成小步骤        按步骤实施        展示证据
```

每步骤的要点：

| 步骤 | 做什么 | 为什么必须做 |
|------|--------|------------|
| brainstorming | 动手前先明确要做什么、有几种方案、选哪个 | 防止方向错误——代码写得再好，方向错了也白写 |
| writing-plans | 把方案拆成可独立执行的小步骤，每步写清楚改哪个文件 | 保证可跟踪——每步做没做、做得对不对一眼能看出来 |
| executing-plans | 按步骤实现，每步完成后立即验证 | 保证早期发现错误——不要全部写完再发现第 1 步就错了 |
| verification | 展示测试通过、Lint 干净、Build 成功的证据 | 保证不是"嘴炮完成" |

这不是约束 AI 的自由——这是把资深工程师的工作习惯写成流程，让 AI 照着走。

## 三、心智转变的三个触发动作

### 动作 1：每次派活之前先补信息

启动任何 AI 任务前，花 30 秒自问三个问题：

1. 这个任务里的关键术语，AI 能从开源社区文档里弄明白吗？→ 不能就补 CONTEXT.md 引用
2. 涉及的文件范围，AI 能从指令里直接看出来吗？→ 不能就在指令里写清楚
3. 我有没有告诉 AI 哪些文件/模块绝对不能碰？→ 没有就在指令里加边界

### 动作 2：接受"AI 会犯错"并建立纠偏机制

AI 犯错不是系统缺陷，是系统特性。就像新员工犯错不可能避免，但这种犯错的可控性很高——成本低（回滚就是）、传播范围窄（只在本会话里）、可追溯（都有日志）。

建立纠偏机制的三个步骤：
1. 犯错 → review 时发现
2. 分析 → 是信息缺失、指令模糊还是工程约束不足
3. 改进 → 补文档/改指令/加 hook

### 动作 3：停止期待 AI"自己悟"

如果指望 AI "自己想明白"你的项目约定，你就会反复失望。这不是 AI 的错——它没有读心术。

转而主动做这件事：假设 AI 什么都不懂，每次需要它懂什么，就说清楚。这个心理转变是所有技术手段的前提。

---

## 四、总结

把 AI 当成新员工来带，本质上是把 AI 从一个"黑箱"变成了一个"可管理的新同事"。这个心智模型会自然地引导你去做那些真正提升协作效率的事——补文档、写清楚指令、建立 SOP、纠偏而非抱怨。

**一句话**：AI 不是神，不是外包，是第一天入职但代码功底扎实的新同事。你怎么带新员工，就怎么带 AI。
