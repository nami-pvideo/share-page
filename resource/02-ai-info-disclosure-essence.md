# 信息披露的本质：从抽象到具体的距离

> 给 AI 写指令就像告诉别人"帮我带只狗回来"——你脑子里想的是邻居家那只温顺的金毛 Lucky，但 AI 可能会带回一只藏獒。差别不在狗，也不在 AI，差别在于你说出了多少细节。本文从信息披露的角度系统性拆解：AI 协作中，什么该说、什么可以省、怎么让"说出来"这件事的回报最高。

---

## 一、原理篇：为什么信息披露是核心

### 1.1 共识的四层光谱

信息从"一个人的脑子里"变成"团队和 AI 都能读取的文档"，经历了一个由模糊到清晰的连续过程。这个过程可以分为四层：

```text
潜意识 ─→ 个人模糊掌握 ─→ 个人充分掌握 ─→ 团队共识
  ↑           ↑              ↑              ↑
连提出者     单人懂但表达    单人想清楚      写进 PRD /
自己都说不清  不清            但没落文档      CONTEXT.md
```

**四层的典型表现**：

- **潜意识**："这个页面的体验还不够好。"——连提出者自己都说不清哪里不好、怎么算好。
- **个人模糊掌握**："把那个按钮置灰。"——可没说是 disabled 还是换个浅色、在什么条件下触发。
- **个人充分掌握**："当用户余额不足时，把'立即支付'按钮置为 disabled，灰度值走设计 Token `--btn-disabled`，保留 hover 提示'余额不足'，禁用 onClick。"——脑子里全想好了，但没说出口、没写下来。
- **团队共识**：上述内容已经写进 PRD、设计稿、或组件库文档，团队所有人（包括 AI）一打开就能读到。

**绝大多数 AI 翻车的根因，是需求停留在前两层就甩给 AI 了**。工程师脑子里有一张完整的图——这在它的脑子里是一个"充分掌握的清晰画面"——但说出来的话却只是"模糊掌握"级别的几个词。AI 拿到的是一张信息严重缺损的指令，它只能用世界知识去"脑补"缺失的部分。这种脑补，就是翻车的源头。

### 1.2 "狗"与"你家那只金毛"——抽象与具体的距离

用一组不严格但传神的类比来理解这个问题：

- "狗"——一个抽象概念。它有四条腿、会叫、有毛。但具体长什么样、多大、什么性格，完全不确定。
- "金毛"——具体了一层。你知道它大概 25-35 公斤、金黄色、性格温顺、爱流口水。
- "邻居家那只叫 Lucky 的金毛"——这才是最具体的。你知道它多少斤、喜欢什么、害怕什么。

编程需求遵循完全相同的规律：

| 抽象级别 | 类比 | 编程中的对应 |
|---------|------|------------|
| 品种泛指 | "改一下这个页面" | 界面改动、UI 层面的一句话 |
| 个体具体 | "余额不足时把'立即支付'置为 disabled，灰度走 Token --btn-disabled，hover 提示'余额不足'" | 精确到 Token、状态、交互行为的规格 |

说"把按钮置灰"，AI 实现的是"一只标准金毛"——你拿到的是一个普遍意义上的灰色按钮，但不是你脑子里那个。你需要说清楚 token 名称、disabled 状态、hover 文案，AI 才能交付你想要的。

**抽象到具体的距离，就是信息披露的工作量**。距离越远，需要补充的信息越多。

### 1.3 为什么跟人协作不需要这么具体的披露

一个自然的疑问：跟同事协作时，"把按钮置灰"就够了，为什么跟 AI 协作要补这么多？

答案是：**同事有团队共识兜底**。

- 同事知道项目用的设计系统
- 同事知道 disabled 按钮的大致样式规范
- 同事知道什么场景下按钮需要 disabled
- 如果同事不知道，他会主动追问

AI 没有这层兜底。它如果不知道你们的设计 Token 名字，就会自己发明一个；如果不知道什么情况下按钮应该 disabled，就会按"常见做法"猜一个。它不会追问——它会直接做，而且做出来的解释还很有道理。

**信息披露的核心意义**：把"AI 替你猜"替换成"AI 照着做"，把猜测带来的不确定性消解在源头。

## 二、基础设施篇：信息披露依赖什么工具

### 2.1 Agent 模式下的自主扩散机制

要理解信息披露为什么影响这么大，必须先理解 AI 编码工具的工作模式。

市面上的 AI 编码工具（Claude Code、Cursor、GitHub Copilot Chat、Devin 等）绝大多数运行在 **Agent 模式**下：

```
[接收目标] → [思考] → [调用工具: grep/read/bash] → [拿到信息]
              ↑                                          │
              └──────── [继续思考/决策] ←────────────────┘
                                  ↓
                          [输出代码或回答]
```

关键特征：**AI 会自己去搜集信息**。你说"优化错误处理"，它会自动 grep "error" 关键词，读它认为相关的文件，然后改它认为需要改的代码。

这有两面性：

- **好的一面**：你不用把每个文件都手动喂给它，省了大量体力。
- **坏的一面**：它的"搜索"可能触及不该碰的模块，它的"判断"可能误判相关性。在指令模糊的情况下，它会扩散——因为它默认"找到的都是相关的"。

### 2.2 Workflow 模式下的固定流水线

另一种模式是 **Workflow**，常见于 n8n、Dify、Make、Airflow 等工具：

```text
[触发] → [步骤1: 取数据] → [步骤2: 调用模型] → [步骤3: 写入库]
         必填: api_key       必填: prompt          必填: db_conn
```

每一步的输入必须前置提供。缺一个参数就跑不动。没有"自主补全"的能力。

对比：

| 维度 | Workflow | Agent |
|------|----------|-------|
| 信息要求 | 一步不缺，缺就报错 | 可以缺，缺了自己找 |
| 出错模式 | 明确报错 | 静默走偏，不易察觉 |
| 适合任务 | 重复性、输入输出固定 | 探索性、边界模糊 |
| 代表工具 | n8n、Dify、Airflow | Claude Code、Cursor、Devin |

关键洞察：**Agent 模式的"自主扩散"特性决定了信息披露的质量直接决定代码输出的质量**。信息披露无效率 = Agent 自由扩散 = 改不该改的代码。

### 2.3 业界工具中的信息披露实践

**CLAUDE.md / AGENTS.md 机制**：

这是最主流的持久信息注入方式。在项目根目录放置一个 Markdown 文件，AI 工具在每次会话开始时会自动读取。

Claude Code 的 CLAUDE.md、Codex 的 AGENTS.md、Cursor 的 .cursorrules、GitHub Copilot 的 .github/copilot-instructions.md——它们本质上是同一个模式的不同实现。

典型的 CLAUDE.md 内容结构：

```markdown
## Architecture
| Layer | Location | Rule |
| --- | --- | --- |
| Handler | handlers/ | HTTP binding only, no business logic |
| Service | services/ | Orchestration, transactions |
| Repository | repositories/ | Database access only |

## Rules
- Prefer the smallest change that solves the issue
- Do not refactor unrelated code
- Match existing naming conventions
```

这本质上是通过工程手段，把"团队共识"层持久化注入到每次 AI 会话中。

**MCP（Model Context Protocol）**：

Anthropic 推出的开放协议，让 AI 工具在运行时通过标准化接口拉取外部信息。它解决的是"不确定 AI 会不会用到"的信息——比如 API 文档、数据库 schema、配置中心的规则。AI 在需要时才拉，不预占上下文窗口。

**代码库索引与搜索**：

Cursor 的 Codebase Indexing、Continue 的 Codebase RAG、Sourcegraph Cody 的代码库理解——这些工具通过对项目代码建索引，让 AI 不逐文件读取也能定位相关代码。它们在 Agent 的"自主探索"阶段提供高效的信息检索能力，减少 token 消耗和走偏风险。

## 三、实践篇：提高信息披露的效率

### 3.1 最小信息披露原则

不是"把所有信息都告诉 AI"，而是"只告诉 AI 它不能自己推断出来的信息"。

可以快速判断的原则：

| AI 自己能推断出来的 | 不需要披露 | 例子 |
|-------------------|----------|------|
| Go 标准库的用法 | 不需要 | `context.WithTimeout` |
| 主流框架的常见用法 | 不需要 | Gin 怎么挂中间件 |
| 通用设计模式 | 不需要 | Repository 模式怎么用 |

| AI 自己推断不出来的 | 必须披露 | 例子 |
|-------------------|---------|------|
| 项目特定的命名约定 | 必须 | 文件名用 snake_case |
| 项目特定的架构约束 | 必须 | Service 不能调 Handler |
| 项目特有的术语定义 | 必须 | "作业"和"任务"是不同的概念 |
| 本次需求的范围和限制 | 必须 | 只改 Provider 层 |

### 3.2 三条披露通道

实践中，主要的信息披露通道有三条：

| 通道 | 载体 | 时效 | 用途 |
|------|------|------|------|
| 长期约束文件 | CLAUDE.md / AGENTS.md | 持久，会话启动时自动读 | 架构分层、命名规范、禁止事项 |
| 领域术语文件 | CONTEXT.md | 持久，需手动引用 | 消除术语歧义，统一 AI 的概念空间 |
| 本次需求指令 | 对话输入 | 一次性 | 文件范围、验收标准、明确边界 |

**通道 1 和通道 2 的核心价值是"减少每次需求的披露成本"**。当 CLAUDE.md 已经把"Handler 不能写业务逻辑"写死了之后，每次派任务就不需要再说一次"不要往 Handler 里塞业务逻辑"。

**通道 3 是每次协作都绕不开的**。关键不是写多长，而是包含三个要素：起点（从哪个文件开始看）、边界（哪些不能动）、验收方式（怎么算完成）。

### 3.3 披露效率的优化方向

几个提高信息披露投入产出比的策略：

**策略一：用术语表消除重复解释**。维护一份 CONTEXT.md 领域术语表。每次遇到 AI 把两个概念搞混，就在术语表里补一条。下次说"参考 CONTEXT.md 的 XX 定义"远比每次手动解释高效。

**策略二：在 CLAUDE.md 里写 Do/Don't，不是抽象原则**。"保持关注点分离"——AI 理解这句话，但它不知道在你的项目里具体怎么落地。写成"Handler 层只做 HTTP 绑定，不含业务逻辑"——可被执行。

**策略三：持续累积需求文档**。不要每次新需求都另起一个 PRD 文件，而是在已有文档上追加。这会让同一个模块的历史决策、边界条件、过往坑点自然累积。下次再碰这个模块，AI 一份文档就拿到了完整上下文。

## 四、总结

信息披露是 AI 协作中"投入小、回报大"的关键动作。它不是多写文档的问题，而是把已经在脑子里的东西，花 30 秒用 AI 能理解的方式说出来。

**一句话**：信息说不清，AI 替你猜；猜对了是你运气好，猜错了是你 review 的工时。下次给 AI 派活前，问自己一句——它在什么地方需要猜？
