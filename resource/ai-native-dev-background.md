# AI Native 文档族 · 决策背景

> 本文档记录 `visual_gen` 仓库中 AI Native 主题文档族的创作背景、关键决策与维护原则。它不教读者怎么用 AI，而是回答两个问题：**「这几份文档为什么是现在这个样子？」**和**「以后要新增内容应该补到哪一份？」**

---

## 一、文档族构成

仓库 `docs/` 目录下目前有四份相互关联的 AI Native 主题文档，加上一份配套设计 spec，共五份：

| 文档 | 路径 | 体量 | 角色 |
|------|------|------|------|
| 流程 SOP | `docs/ai-native-dev-flow.md` | ~26K，6 阶段流程 | HOW · 标准作业流程，作为团队执行手册 |
| 分享讲稿 | `docs/ai-native-sharing.md` | ~7K，45–60 分钟讲稿 | WHY · 入门心法分享，作为讲稿+独立读物双用途 |
| 深度文章 | `docs/ai-native-dev-info.md` | ~12K，微信公众号风格 | WHY 加强版 · 深度扩写，作为对外发布材料 |
| 决策背景 | `docs/ai-native-dev-background.md` | 本文 | META · 文档族决策记录，作为维护者参考 |
| 设计 spec | `docs/superpowers/specs/2026-06-11-ai-native-sharing-design.md` | ~5K | DESIGN · 分享讲稿的创作设计书 |

四份正文文档构成一个完整的认知阶梯：分享讲稿用「类比 + 案例」打开直觉，深度文章用「概念 + 表格 + 模板」补全细节，流程 SOP 用「阶段 + 门禁 + skill 清单」指导落地，决策背景做横向维度的元说明。

## 二、创作起点与受众

### 2.1 起点：一份 Obsidian 笔记 + 已有的 SOP

整个文档族的种子是一份个人 Obsidian 笔记，其中提出了几个核心论点：

- **AI 的核心约束在于信息流转**——AI 不会读心术，每次会话都是从零开始；
- 信息可分为**世界知识 / 团队共识 / 当前数据**三层，AI 默认只有第一层；
- 需求的清晰程度可分为**潜意识 / 个人模糊 / 个人充分 / 团队共识**四个层次；
- **抽象与具体**的距离（「狗」与「你家那只金毛」）映射到需求时同样存在。

笔记结尾还列出五个团队级流程的延伸：文档中心流转、harness 工程、agent vs workflow、git worktree、superpower 流程。

仓库里此前已经有一份 `ai-native-dev-flow.md`——一份高密度的 SOP，从 Phase 0 需求沉淀讲到 Phase 5 文档沉淀，配 AI 提效模型、skill 速查表、项目配置清单。这份文档强在「怎么做」，弱在「为什么这么做」。新加入的同学翻开就是六个阶段、八张表格、几十个 skill 名字，难免被冲懵。

### 2.2 受众画像

要服务的核心受众是**懂代码、少用 AI 的内部开发同事**。这个画像有几个关键特征：

- **编程基本功扎实**：可以直接使用仓库术语（Provider、Gateway、Registry、Repository），不需要补编程基础；
- **AI 实战经验少**：需要补 AI 侧术语（context、prompt、skill、agent、harness、workflow、hooks）；
- **能动手能 Review**：不是单向接受信息的「观众」，而是会上手用 AI 的协作者；
- **没耐心读纯理论**：需要类比和实战案例来锚定概念，不是论文式的定义堆砌。

围绕这个画像，文档族选择**类比驱动 + 仓库实物 + 反面案例**作为主轴，而不是纯理论框架。

## 三、为什么是三份正文，而不是一份

最初的设想是「在 `ai-native-dev-flow.md` 前面加一段 WHY」，但这条路被否决，理由有三：

1. **混合文档既不适合讲稿，也不适合工具书**。SOP 文档追求查找效率（看目录直接跳到 Phase 3），讲稿追求叙事节奏（开场要钩住听众）。两套需求强行揉到一份里，最后会变成「前面是散文后面是说明书」，两头不靠。
2. **WHY 篇和 WHY 加强版需要不同的语态**。讲稿用第一人称叙事，听众会感觉「这是分享者真实踩过的坑」；公众号风格的深度文章用客观第三人称，读者会感觉「这是经过提炼的方法论」。同一份文档无法同时承载两种语态。
3. **不同读者的入口期望不同**。新人想要 30 分钟读懂、决策者想要纵深、执行者想要 SOP。让单一文档承担三种入口职责，每一类读者都会觉得「太长」或「太浅」。

最终方案是**让每份文档在自己的车道上跑**：

```
新同事入职第一天        分享讲稿 (sharing.md)        45–60 分钟建立直觉
                            ↓
准备技术分享/对外发布    深度文章 (dev-info.md)      微信公众号长文，深度扩写
                            ↓
日常开发执行            流程 SOP (dev-flow.md)       Phase 0–5 + skill 速查
                            ↓
文档维护者              决策背景 (本文)              横向元说明，避免重复造轮子
```

## 四、关键创作决策

### 4.1 决策一：用「AI = 新员工入职」作为类比锚点

候选类比有三个：

1. **新员工入职**——首日报到的工程师，会编程但不懂项目；
2. **外包合作**——能交付但需要明确边界；
3. **实习生**——会犯错但成长快。

最终选「新员工入职」的理由：

- **新员工**这个身份把 AI 从「神」或「外包」的两极拉回中间，正好对应受众需要建立的心理预期；
- **入职**这个动作天然包含「资料包 / 工位 / SOP / 师傅带教」等一系列具体环节，能逐一映射到文档族要讲的内容；
- 这个类比对**任何在职开发同事都不陌生**，零成本理解。

候选 2（外包）的问题：暗示「甩需求」式协作，与文档想倡导的「指路而非堆要求」相左；候选 3（实习生）的问题：暗示能力有限，与受众日常用到的 Claude Code 实际能力不符。

类比一旦确定，全文都围绕它展开：
- 信息分层 → 入职第一天的资料差距
- 信息披露 → 给新员工指路的方式
- 上下文管理 → 决定新员工今天看什么、不看什么
- 文档中心 → 入职资料包
- harness → 工位与工具栏
- git worktree → 每个任务一张干净桌子
- superpower → 资深同事的工作 SOP

### 4.2 决策二：心法 60% + 5 流程 40% 的权重

Obsidian 笔记里有两块内容：**信息披露 + 上下文管理（心法）**和**五个团队流程（落地）**。两者权重如何分配？

候选方案：

| 方案 | 心法 | 流程 | 取舍 |
|------|------|------|------|
| A | 90% | 10% | 心法深讲，流程只做导读 |
| B | 60% | 40% | 心法主线，流程作落地印证 |
| C | 50% | 50% | 上下半部独立章节 |
| D | 30% | 70% | 以流程为骨架，心法贯穿 |

选 B 的理由：

- **受众是「懂代码、少用 AI」的开发**，他们日常不缺工程经验，缺的是 AI 协作的认知模型。心法的边际收益高于流程介绍。
- 但**纯心法没有落地，听众散场就忘**。五个流程作为「这些心法在我们项目里长什么样」的具体印证，能把抽象概念落到他们今天就能动手的实践上。
- **60/40** 这个比例正好对应 45 分钟讲稿里 25 分钟心法 + 15 分钟流程 + 5 分钟开场收尾，节奏合理。

方案 A 会让流程变成「然后……」的草草带过；方案 D 会让心法变成五个流程章节的附庸，无法独立站立。

### 4.3 决策三：agent vs workflow 的位置变迁

最初设计里，「agent vs workflow」是 5 流程章节的 section 3——独立讲两种 AI 工具的本质区别。但创作过程中发现：**这个对比不止是术语介绍，它正好可以充当「信息披露为什么关键」的核心论据**。

调整后的逻辑链：

```
workflow 模式 → 每一步输入必须完备 → 缺信息 = 跑不动
agent 模式   → 缺信息会自主搜索   → 但搜索消耗 context、可能找错
                ↓
信息披露的关键不是「全塞」，而是「指对路」
                ↓
披露太少 → agent 自由扩散（顺手重构无关代码）
披露太多 → context window 被噪音占满
```

把这个对比提升到心法 2 之后：

- 信息披露这个抽象概念**立刻有了物理学解释**（agent 的自主搜索机制决定了「指对路」是关键）；
- 5 流程的原 section 3 也得到了**功能重定位**——不再是概念介绍，而是「实操视角：何时各用一种」，与心法 2 的「为什么这个对比重要」互补；
- 整体没有重复，反而让两个章节互相印证。

这个调整是创作过程中最关键的一次结构优化，也是用户在 brainstorming 环节的直接贡献。

### 4.4 决策四：反面案例只用一次

文档族里有一个反面案例——**「让 AI 优化 Provider 错误处理 → 它顺手改了 Gateway 和 Hook」**。它在三份正文里都出现，但出现方式各有不同：

| 位置 | 用途 | 出现形式 |
|------|------|----------|
| `ai-native-sharing.md` 开场 | 钩住听众 | 第一人称叙事，1 次 |
| `ai-native-sharing.md` 心法 2 落地段 | 论据 | 回到这个案例做物理学解释 |
| `ai-native-dev-info.md` 引子（场景 B） | 客观对照 | 第三人称呈现，去掉「我」字 |
| `ai-native-dev-info.md` 心法 2 反面段 | 论据 | 同上 |

为什么不堆更多反面案例？

- **反面案例用多了，文档基调会偏负面**——读者读完会觉得「AI 不靠谱」，与文档想倡导的「AI 是新员工，靠谱与否取决于你怎么带」相反。
- **同一个案例多次回声**比多个案例一次性堆叠更有记忆点。第一次出现是「问题」，第二次出现是「答案」，读者会有「原来开场那个翻车现场是这么回事」的恍然感。
- 数据点：1 个反面 + 1 个正面（加 Provider 一次过）已经能把对比讲透，多了就是噪音。

### 4.5 决策五：仓库实物引用清单

文档族刻意大量引用 `visual_gen` 仓库的真实模块。引用清单：

| 引用对象 | 出处 | 用途 |
|---------|------|------|
| `CLAUDE.md` 的 Surgical Changes 规则 | 仓库根目录 | 心法 2 概念段呼应 |
| `CONTEXT.md` 术语（任务/作品/积分/`function_id`） | 仓库根目录 | 心法 1 类比段、5 流程 section 1 |
| Provider 注册机制（`init()` + alias） | `internal/provider/`、`cmd/api/provider.go` | 心法 3 落地段（正面案例） |
| `batch_coordinator` 父子任务汇总 | `internal/service/batch_coordinator*.go` | 心法 3 钩子段 |
| `ProviderRegistry` 别名解析 | `internal/registry/` | 心法 3 落地段 |
| `detectProviderFailure` 错误检测 | `internal/gateway/zhinao.go` | 心法 2 反面案例 |
| `ErrorClassifier` 可选接口 | `internal/provider/` | 心法 3 落地段细节 |

为什么必须用真实模块，而不是构造抽象例子？

- **抽象例子有「翻译损耗」**——读者要先把例子映射到自己的项目，才能用。真实模块的例子读完就能用。
- **真实例子有「自我验证」效果**——读者随手 grep 一下就能验证文档没胡说，这种「可验证性」直接构成文档可信度。
- **真实例子有「持续维护」压力**——一旦代码改了，文档需要同步。这种压力反而保证文档不会陈旧（陈旧的引用立刻暴露）。

代价是文档跨项目复用性下降。但本文档族的受众就是 `visual_gen` 团队本身，跨项目复用不是首要目标。

### 4.6 决策六：风格选择 —— 第一人称叙事 vs 客观第三人称

`ai-native-sharing.md` 用第一人称：「**我**让 Claude Code……」「**我**盯着 diff 看了二十分钟」。`ai-native-dev-info.md` 全去掉了「我」字，改用「**开发者**」「**实践中**」「**场景 A**」。

为什么不统一？

| 维度 | 分享讲稿 | 深度文章 |
|------|---------|---------|
| 阅读场景 | 现场讲 + 团队内自学 | 微信公众号、对外发布 |
| 期待语态 | 真实经验感（「我踩过」） | 提炼方法论感（「实践表明」） |
| 信任建立路径 | 通过亲身故事 | 通过严谨结构 |
| 第一人称作用 | 拉近距离 | 反而显得不严谨 |

**第一人称在讲稿里是优点**：听众会感觉这不是空谈，是讲者真实踩过的坑，可信度立刻拉满。
**第一人称在公众号文章里是缺点**：读者并不认识作者，「我」字反而让方法论显得像个人感受，而不是普适经验。

所以同一套内容在两份文档里换了语态：
- `sharing.md`：「**我**告诉它『参考 Gemini provider 的写法』」
- `dev-info.md`：「**开发者**告诉它『参考 Gemini provider 的写法』」 / 「**场景 A 的指令**包含明确的参考实现路径」

这个差异在创作时极易被忽略，但对最终呈现效果有显著影响。

### 4.7 决策七：「讲稿 + 独立读物」二合一

`ai-native-sharing.md` 同时承担两种使用场景：

- **现场讲稿**：分享者照着讲，听众边听边看；
- **独立读物**：没参加分享的同事自学。

这要求文档**段落叙事为主，关键节点用表格/列表锚定**。如果纯散文，讲稿不好讲（找不到节奏点）；如果纯列表，独立阅读卡顿（没有上下文衔接）。最终的折中是：

- 每个心法 4 段式（钩子 → 概念 → 类比 → 落地），段落里有连贯叙事；
- 段落内部嵌入 1–2 个表格作为视觉锚点（信息三层、4 层共识、context 三种喂入方式）；
- 5 流程章节用表格组织（讲稿快速带过 / 自学顺序读完）；
- 收尾「三件事」用编号列表（讲稿收束、自学带走）。

## 六、维护原则

### 6.1 新内容该补到哪一份？

```dot
graph "新内容定位决策树" {
    "新内容产生" [shape=oval];
    "类型?" [shape=diamond];

    "新内容产生" -> "类型?";
    "类型?" -> "ai-native-dev-flow.md" [label="阶段/skill 变化"];
    "类型?" -> "ai-native-sharing.md" [label="新的心法/案例"];
    "类型?" -> "ai-native-dev-info.md" [label="对外深度扩写"];
    "类型?" -> "ai-native-dev-background.md (本文)" [label="结构/决策变更"];
}
```

具体准则：

- **新增 skill / 阶段优化** → 补到 `ai-native-dev-flow.md` 的对应 Phase 或 skill 速查表；
- **新的正面/反面案例** → 补到 `ai-native-sharing.md`，**优先替换旧案例而不是堆叠**（保持讲稿时长可控）；
- **对外发布需要的深度细节** → 补到 `ai-native-dev-info.md`，可以扩展模板、矩阵、反模式清单；
- **文档族本身的结构调整** → 补到本文，记录决策；
- **CONTEXT.md 术语变化** → 同时检查三份文档里相关术语是否需要同步更新。

### 6.2 三份正文文档的同步策略

**不要镜像同步**。三份文档故意保持差异：

- 流程 SOP 加了一个新 skill，不一定要在讲稿里讲——讲稿只覆盖最核心的 4 个 skill 主线；
- 讲稿加了一个新案例，深度文章可以扩写成完整反模式分析，但流程 SOP 不必动；
- 深度文章可以独立演进对外发布的需要，不影响内部 SOP。

**同步触发点只有两个**：

1. **核心心法发生变化**（例如发现「信息披露」需要拆成更细的子概念）——三份文档都要更新；
2. **仓库引用的实物文件被改名/删除**——三份文档都要校对。

### 6.3 退役与归档

`ai-native-sharing.md` 是相对易过时的文档：

- 它绑定了一次具体的分享场景；
- 它依赖具体的仓库代码现状作为案例。

如果未来文档族扩展出针对不同受众的多份讲稿（例如「面向 PM」「面向跨部门」），原讲稿可以重命名归档（`ai-native-sharing-dev-2026.md`），保留为历史版本。

`ai-native-dev-flow.md` 是最稳定的文档——它的 SOP 框架反映团队工作流，只要工作流不大改，文档就持续生效。

`ai-native-dev-info.md` 介于两者之间，建议每次对外发布前重审一遍仓库实物引用的准确性。

## 七、后续演进路径

文档族已落地的部分到 2026-06 截止。未来可以扩展的方向：

| 方向 | 形态 | 触发条件 |
|------|------|---------|
| Skill 实操手册 | 每个 skill 独立的最小示例集 | 团队成员对某个 skill 反复问「具体怎么用」 |
| 反模式专题 | 收集真实翻车案例 + 根因 + 修复 | 出现 3 次以上同类问题 |
| 多角色版本 | PM / 设计 / QA 各自视角的简版 | 跨职能协作需求出现 |
| 对外公开版 | 去敏感化后投稿技术社区 | 团队判定有对外影响价值时 |

这些扩展不在当前文档族范围内，但本文档结构预留了它们的位置——未来新增任一方向，都可以在本文「文档族构成」表中加一行，并在「维护原则」中补一条决策依据。
